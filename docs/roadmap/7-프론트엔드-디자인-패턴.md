# 7. 프론트엔드 디자인 패턴

## 목적

프론트엔드에서 자주 쓰이는 디자인 패턴을 문제 해결 관점으로 정리한다.

디자인 패턴은 코드를 멋있게 보이게 만드는 이름표가 아니라, 반복되는 설계 문제를 해결하기 위한 선택지다. 패턴을 적용하기 전에 먼저 어떤 문제를 풀려는지 명확히 해야 한다.

## 핵심 문장

패턴은 목적이 있을 때만 사용한다.

로직 재사용, UI 자유도, 상태 공유, 의존성 분리, 부수효과 격리처럼 해결하려는 문제가 명확하지 않다면 패턴을 적용하지 않는다.

## 패턴 선택 기준

| 문제 | 고려할 패턴 |
| --- | --- |
| 로직은 재사용하고 UI는 호출부가 결정해야 한다 | Render Props, Headless Component |
| 여러 하위 컴포넌트가 하나의 상태를 공유해야 한다 | Compound Component + Context |
| 성공/실패 알림 같은 부가 동작을 action에 붙여야 한다 | Decorator / Higher-order Function |
| 구현 세부사항을 외부에서 주입해야 한다 | Dependency Injection |
| 앱 전체에서 하나의 인스턴스만 공유해야 한다 | Singleton |
| 서로 직접 알면 안 되는 모듈 간 이벤트를 전달해야 한다 | Pub/Sub |
| 여러 구현 중 하나를 선택해야 한다 | Strategy |
| 외부 DTO/API 응답을 내부 모델로 바꿔야 한다 | Adapter / Mapper |
| 복잡한 분기 로직을 명시적 상태 전이로 관리해야 한다 | State Machine |

## 1. Render Props Pattern

Render Props는 로직을 가진 컴포넌트가 직접 UI를 그리지 않고, 렌더링 책임을 외부에서 받은 함수에게 위임하는 패턴이다.

React에서 로직과 렌더링을 분리하기 위한 컴포넌트 합성 패턴이며, Headless UI를 구현하는 대표적인 방식 중 하나다.

### 핵심 구조

```md
1. 공통 로직을 가진 제공자 컴포넌트가 있다.
2. 제공자 컴포넌트는 상태와 action을 만든다.
3. UI는 직접 결정하지 않는다.
4. children/render 함수에 상태와 action을 넘긴다.
5. 호출부가 UI를 렌더링한다.
```

### 적합한 경우

- 로직은 공통이다.
- UI는 케이스마다 다르다.
- props 옵션으로 UI 변형을 늘리면 컴포넌트가 복잡해진다.
- Context를 만들기엔 범위가 좁다.
- hook만으로는 렌더링 영역을 제한하기 어렵다.

### Do

```tsx
const RowSelection = <Row extends { id: string | number }>({
  rows,
  children,
}: {
  rows: Row[];
  children: (params: {
    selectedRows: Row[];
    selectedIds: Array<string | number>;
    toggleRow: (row: Row) => void;
    clearSelection: () => void;
  }) => React.ReactNode;
}) => {
  const [selectedIds, setSelectedIds] = useState<Array<string | number>>([]);

  const toggleRow = (row: Row) => {
    setSelectedIds((prev) =>
      prev.includes(row.id)
        ? prev.filter((selectedId) => selectedId !== row.id)
        : [...prev, row.id],
    );
  };

  const clearSelection = () => setSelectedIds([]);

  const selectedRows = rows.filter((row) => selectedIds.includes(row.id));

  return <>{children({ selectedRows, selectedIds, toggleRow, clearSelection })}</>;
};
```

```tsx
<RowSelection rows={rows}>
  {({ selectedRows, selectedIds, toggleRow, clearSelection }) => (
    <>
      <OutboundToolbar
        selectedCount={selectedRows.length}
        onClearSelection={clearSelection}
      />
      <OutboundTable
        rows={rows}
        selectedIds={selectedIds}
        onToggleRow={toggleRow}
      />
    </>
  )}
</RowSelection>
```

### Do Not

```tsx
<RowSelection
  rows={rows}
  variant="outbound"
  showToolbar
  showClearButton
  toolbarPosition="top"
  tableType="baseGrid"
  onRenderActionButtons={renderActionButtons}
/>
```

UI 변형을 props 옵션으로 계속 늘리는 대신 render props로 렌더링을 호출부에 맡긴다.

### CGKR 적용 후보

- `ExcelUploadController`
- `RowSelectionController`
- `BarcodeInputController`
- `PermissionAction`
- `PrintController`

### 피해야 하는 경우

- 단순 상태 하나만 재사용한다.
- API 호출과 mutation orchestration을 감싸려 한다.
- 출고 확정, 포장 완료 같은 복잡한 업무 flow를 숨기려 한다.
- 여러 하위 컴포넌트가 넓은 범위에서 상태를 공유해야 한다.

단순 상태는 hook이 낫고, 넓은 상태 공유는 Context/Compound Component가 낫다.

## 2. Headless Component Pattern

Headless Component는 상태와 행동은 제공하지만 스타일과 마크업은 강제하지 않는 컴포넌트다.

Render Props, hook, Compound Component 모두 headless UI를 구현하는 방식이 될 수 있다.

### Do

```tsx
const Toggle = ({
  children,
}: {
  children: (params: { checked: boolean; toggle: () => void }) => React.ReactNode;
}) => {
  const [checked, setChecked] = useState(false);

  return <>{children({ checked, toggle: () => setChecked((prev) => !prev) })}</>;
};
```

```tsx
<Toggle>
  {({ checked, toggle }) => (
    <button onClick={toggle}>
      {checked ? "ON" : "OFF"}
    </button>
  )}
</Toggle>
```

### Do Not

```tsx
const Toggle = ({ checkedLabel, uncheckedLabel, color, size, rounded }) => {
  // 모든 UI 변형을 props로 처리하려고 한다.
};
```

Headless 패턴은 UI 변형을 컴포넌트 내부 옵션으로 늘리지 않고 호출부에 맡긴다.

## 3. Compound Component Pattern

여러 하위 컴포넌트가 하나의 상태를 공유해야 할 때 사용한다.

### 적합한 경우

- `Tabs`, `Dropdown`, `Menu`, `Modal`처럼 구조가 있는 UI다.
- 여러 하위 컴포넌트가 같은 상태를 공유한다.
- props drilling 없이 선언적인 조합이 필요하다.

### Do

```tsx
<Tabs defaultValue="order">
  <Tabs.List>
    <Tabs.Trigger value="order">주문</Tabs.Trigger>
    <Tabs.Trigger value="invoice">송장</Tabs.Trigger>
  </Tabs.List>

  <Tabs.Panel value="order">
    <OrderPanel />
  </Tabs.Panel>

  <Tabs.Panel value="invoice">
    <InvoicePanel />
  </Tabs.Panel>
</Tabs>
```

### Do Not

```tsx
<Tabs
  defaultValue="order"
  tab1Label="주문"
  tab1Content={<OrderPanel />}
  tab2Label="송장"
  tab2Content={<InvoicePanel />}
  layout="horizontal"
  activeColor="blue"
/>
```

구조적 UI를 props 옵션으로만 제어하면 확장할수록 복잡해진다.

## 4. Decorator / Higher-order Function Pattern

기존 action에 성공/실패 알림, 로깅, 권한 체크 같은 부가 동작을 붙일 때 사용한다.

### Do

```ts
export const withTryNotifySuccess = <Args extends readonly unknown[], Result>(
  action: (...args: Args) => Promise<Result>,
  successMessage: string,
  addNotification: (message: string, type: "success" | "error") => void,
  getErrorMessage: (error: unknown) => string,
) => {
  return async (...args: Args): Promise<Result | undefined> => {
    try {
      const result = await action(...args);
      addNotification(successMessage, "success");
      return result;
    } catch (error) {
      addNotification(getErrorMessage(error), "error");
      return undefined;
    }
  };
};
```

```ts
const handleScan = withTryNotifySuccess(
  scanPackingItem,
  "스캔 처리되었습니다.",
  addNotification,
  getPackingErrorMessage,
);
```

### Do Not

```tsx
const handleScan = async () => {
  try {
    await scanPackingItem();
    addNotification("스캔 처리되었습니다.", "success");
  } catch (error) {
    addNotification(getPackingErrorMessage(error), "error");
  }
};
```

반복되는 성공/실패 처리 로직이 여러 컴포넌트에 흩어진다.

## 5. Adapter / Mapper Pattern

외부 데이터 형태를 내부 모델로 변환할 때 사용한다.

### Do

```ts
const mapOrderResponseToOrder = (response: OrderResponse): Order => {
  return {
    id: response.ord_no,
    status: response.ord_status,
    invoiceNo: response.inv_no ?? null,
  };
};
```

```tsx
const OrderStatusBadge = ({ status }: { status: OrderStatus }) => {
  return <Badge>{status.label}</Badge>;
};
```

### Do Not

```tsx
const OrderStatusBadge = ({ order }: { order: OrderResponse }) => {
  return <Badge>{order.ord_status === "10" ? "주문 접수" : "출고 준비"}</Badge>;
};
```

서버 DTO를 컴포넌트까지 끌고 오지 않는다.

## 6. Strategy Pattern

여러 정책 중 하나를 외부에서 주입해야 할 때 사용한다.

### Do

```ts
const orderByHighPrice = (a: Product, b: Product) => b.price - a.price;
const orderByLowPrice = (a: Product, b: Product) => a.price - b.price;

const sortProducts = (
  products: Product[],
  orderBy: (a: Product, b: Product) => number,
) => {
  return [...products].sort(orderBy);
};
```

```ts
const products = sortProducts(rawProducts, orderByHighPrice);
```

### Do Not

```ts
const sortProducts = (products: Product[], sortType: "high" | "low") => {
  if (sortType === "high") {
    return [...products].sort((a, b) => b.price - a.price);
  }

  return [...products].sort((a, b) => a.price - b.price);
};
```

분기가 늘어나는 정책은 함수로 주입하는 편이 더 명확할 수 있다.

## 7. Dependency Injection Pattern

상위 로직이 하위 구현 세부사항에 직접 의존하지 않게 할 때 사용한다.

### Do

```ts
const createDownloadAction = ({
  downloadFile,
  saveFile,
}: {
  downloadFile: () => Promise<Blob>;
  saveFile: (blob: Blob) => void;
}) => {
  return async () => {
    const blob = await downloadFile();
    saveFile(blob);
  };
};
```

### Do Not

```ts
const downloadAction = async () => {
  const blob = await apiClient.get("/download");
  saveAs(blob, "download.xlsx");
};
```

테스트나 실행 환경에 따라 구현을 바꿔야 한다면 의존성을 외부에서 주입한다.

## 8. Singleton Pattern

애플리케이션 전체에서 하나의 인스턴스만 공유해야 할 때 사용한다.

프론트엔드에서는 `apiClient`, monitoring reporter, event bus, storage adapter처럼 앱 전역 인프라 객체에 제한적으로 사용한다.

### 적합한 경우

- 앱 전체에서 동일한 설정과 상태를 공유해야 한다.
- 인스턴스가 여러 개 생기면 요청 interceptor, 로깅, 이벤트 구독이 중복된다.
- 도메인 상태가 아니라 인프라 객체다.

### Do

```ts
class ApiClient {
  constructor(private readonly baseURL: string) {}

  get<T>(url: string): Promise<T> {
    return fetch(`${this.baseURL}${url}`).then((response) => response.json());
  }
}

export const apiClient = new ApiClient("/api/basic/v1");
```

```ts
export const fetchOrderList = (params: OrderListRequest) => {
  return apiClient.get<OrderListResponse>("/orders");
};
```

### Do Not

```ts
const fetchOrderList = (params: OrderListRequest) => {
  const apiClient = new ApiClient("/api/basic/v1");

  return apiClient.get<OrderListResponse>("/orders");
};
```

요청마다 client 인스턴스를 만들면 interceptor, auth header, monitoring 설정이 분산될 수 있다.

### 주의

Singleton을 도메인 상태 저장소로 남용하지 않는다.

```ts
export const orderSelection = {
  selectedOrderId: null,
};
```

화면 선택 상태나 임시 UI 상태는 Singleton이 아니라 local state, URL state, Context, store 중 적절한 위치에 둔다.

## 9. Pub/Sub Pattern

발행자와 구독자가 서로 직접 알지 않아야 하는 이벤트를 전달할 때 사용한다.

Pub/Sub는 모듈 간 직접 의존을 줄일 수 있지만, 이벤트 흐름이 숨겨지기 쉬우므로 전역 남용을 피한다.

### 적합한 경우

- 서로 다른 레이어가 직접 import로 엮이면 안 된다.
- 한 이벤트에 여러 구독자가 반응해야 한다.
- 인프라성 이벤트다.
- 이벤트 흐름을 명확한 이름과 타입으로 관리할 수 있다.

### Do

```ts
export type LanguageCode = "ko" | "en" | "ja";
export type Translations = Record<string, string>;
export type Subscriber = () => void;

class LanguageStore {
  private _subscribers: Set<Subscriber> = new Set();
  private _translations: Translations = {};
  private _languageTypeCd: LanguageCode = "ko";

  public getSnapshot = (): Translations => {
    return this._translations;
  };

  public getServerSnapshot = (): Translations => {
    return this._translations;
  };

  public getLanguageCode = (): LanguageCode => {
    return this._languageTypeCd;
  };

  public subscribe = (callback: Subscriber): (() => void) => {
    this._subscribers.add(callback);

    return () => {
      this._subscribers.delete(callback);
    };
  };

  private _emit(): void {
    this._subscribers.forEach((callback) => {
      callback();
    });
  }

  public setTranslations(translations: Translations, languageTypeCd?: LanguageCode): void {
    this._translations = translations;

    if (languageTypeCd) {
      this._languageTypeCd = languageTypeCd;
    }

    this._emit();
  }
}

export const languageStore = new LanguageStore();
```

```ts
export function useTranslation() {
  const translations = useSyncExternalStore(
    languageStore.subscribe,
    languageStore.getSnapshot,
    languageStore.getServerSnapshot,
  );

  const t = (code: string, fallback?: string) => {
    return translations[code] || fallback || code;
  };

  return {
    t,
    translations,
    languageTypeCd: languageStore.getLanguageCode(),
  };
}
```

`languageStore.setTranslations()`가 번역 변경을 publish하고, `useTranslation()`을 사용하는 컴포넌트들이 `languageStore.subscribe`를 통해 변경을 구독한다.

이 구조는 동시에 Singleton이기도 하다. 앱 전체가 하나의 `languageStore` 인스턴스를 공유하고, 그 store의 변경을 여러 컴포넌트가 구독한다.

### Do Not

```ts
languageStore.setTranslations(translations);
window.location.reload();
```

```ts
const Component = () => {
  const langList = JSON.parse(localStorage.getItem("langList") ?? "{}");

  return <span>{langList.WD000105}</span>;
};
```

번역 변경을 전체 새로고침으로 처리하거나, 컴포넌트가 직접 localStorage에서 번역 데이터를 읽으면 React 렌더링과 외부 store 동기화가 깨진다.

### CGKR 적용 후보

- i18n 번역 store 변경 구독
- 로그인 만료 이벤트
- 로컬 프린터 연결 끊김 이벤트
- 공통 API 에러 모니터링 이벤트
- 브라우저 storage/session 동기화 이벤트

### 피해야 하는 경우

- 부모/자식 간 단순 이벤트 전달
- 같은 화면 안의 상태 변경
- query invalidate를 우회하기 위한 이벤트
- 도메인 workflow를 숨기는 이벤트

이런 경우에는 props, Context, React Query invalidate, use-case hook이 더 명확하다.

## 10. State Machine Pattern

복잡한 상태 전이와 분기 로직이 많을 때 사용한다.

### 적합한 경우

- 단계가 명확하다.
- 가능한 상태가 제한되어 있다.
- 어떤 상태에서 어떤 이벤트가 가능한지 명확해야 한다.
- 잘못된 상태 전이를 막아야 한다.

### 예시

```ts
type PackingStep = "SCAN_CART" | "SCAN_ITEM" | "CONFIRM" | "DONE";

const getNextPackingStep = (
  currentStep: PackingStep,
  event: "CART_SCANNED" | "ITEM_SCANNED" | "CONFIRMED",
): PackingStep => {
  if (currentStep === "SCAN_CART" && event === "CART_SCANNED") {
    return "SCAN_ITEM";
  }

  if (currentStep === "SCAN_ITEM" && event === "ITEM_SCANNED") {
    return "CONFIRM";
  }

  if (currentStep === "CONFIRM" && event === "CONFIRMED") {
    return "DONE";
  }

  return currentStep;
};
```

### Do Not

```ts
if (isCartScanned && !isItemScanned && !isConfirmed) {
  // ...
}

if (isCartScanned && isItemScanned && !isConfirmed) {
  // ...
}

if (isCartScanned && isItemScanned && isConfirmed) {
  // ...
}
```

boolean flag 조합이 늘어나면 가능한 상태를 추론하기 어려워진다.

## 패턴 선택 체크리스트

```md
1. 해결하려는 문제가 명확한가?
   -> 아니면 패턴을 적용하지 않는다.

2. 로직은 같고 UI가 다양하게 바뀌는가?
   -> Render Props 또는 Headless Component를 고려한다.

3. 여러 하위 컴포넌트가 같은 상태를 공유하는가?
   -> Compound Component + Context를 고려한다.

4. 성공/실패 알림, 로깅 같은 부가 동작이 반복되는가?
   -> Decorator / Higher-order Function을 고려한다.

5. 외부 DTO를 내부 모델로 바꾸는가?
   -> Adapter / Mapper를 사용한다.

6. 분기 가능한 정책이 계속 늘어나는가?
   -> Strategy를 고려한다.

7. 앱 전체에서 하나의 인스턴스만 공유해야 하는가?
   -> Singleton을 고려한다.

8. 서로 직접 알면 안 되는 모듈 간 이벤트 전달이 필요한가?
   -> Pub/Sub를 고려한다.

9. 상태 전이가 복잡한가?
   -> State Machine을 고려한다.
```

## 금지 패턴

### 패턴 이름을 먼저 정하고 코드를 끼워 맞추기

```md
이건 Strategy로 만들자.
이건 Compound Component로 만들자.
```

문제를 먼저 보고 패턴은 나중에 선택한다.

### 모든 재사용을 Render Props로 해결하기

Render Props는 UI 자유도가 필요할 때 사용한다. 단순 로직 분리는 함수나 hook이 더 단순할 수 있다.

### 모든 공통 UI를 props 옵션으로 확장하기

옵션이 계속 늘어나면 Headless/Render Props/Compound Component를 고려한다.

### 도메인 workflow를 Headless UI에 숨기기

Headless UI는 interaction state에 적합하다. 출고 확정, 송장 발급 같은 업무 flow는 use-case hook으로 분리한다.

### Singleton을 전역 도메인 상태 저장소로 사용하기

Singleton은 인프라 객체에 제한적으로 사용한다. 화면 선택 상태, 검색 조건, 임시 form 상태를 Singleton에 저장하지 않는다.

### Pub/Sub로 명시적 데이터 흐름 우회하기

부모/자식 이벤트, 같은 화면 안의 상태 변경, React Query invalidate를 Pub/Sub로 우회하지 않는다.

## 결론

프론트엔드 디자인 패턴은 React 코드를 복잡하게 만들기 위한 장치가 아니다.

패턴은 로직 재사용, UI 자유도, 상태 공유, 의존성 분리, 부수효과 격리처럼 반복되는 문제를 해결하기 위한 선택지다.

CGKR에서는 UI 자유도가 필요한 작은 headless controller에는 Render Props, 여러 하위 컴포넌트가 상태를 공유하는 구조적 UI에는 Compound Component, 반복되는 action 부가 처리에는 Decorator/Higher-order Function을 제한적으로 사용한다.

앱 전역 인프라 객체에는 Singleton을 제한적으로 사용하고, 서로 직접 알면 안 되는 인프라성 이벤트에는 Pub/Sub를 사용할 수 있다. 다만 도메인 상태와 업무 workflow를 Singleton이나 Pub/Sub 뒤에 숨기지 않는다.
