# 5. 프론트엔드 에러 핸들링 전략

## 목적

프론트엔드에서 발생하는 에러를 종류별로 분류하고, 적절한 계층에서 처리하기 위한 기준을 세운다.

에러 핸들링은 `catch`를 많이 쓰는 것이 아니라 에러의 종류와 처리 책임을 분리하는 것이다.

## 핵심 문장

에러는 발생 위치에서 무조건 처리하지 않는다.

에러를 감지하는 위치와 사용자에게 피드백하는 위치는 다를 수 있다. API layer는 실패를 감지하고, use-case layer는 실패 의미를 해석하며, UI layer는 사용자가 취할 수 있는 행동을 보여준다.

## 에러 분류

| 분류 | 예시 | 주요 처리 위치 |
| --- | --- | --- |
| Network Error | 서버 연결 실패, timeout | `apiClient`, query hook |
| Auth Error | 로그인 만료, 권한 없음 | interceptor, app boundary |
| Validation Error | 입력값 오류 | form validation |
| API Business Error | 재고 부족, 취소 불가 상태 | use-case hook |
| Deserialize Error | Zod schema parse 실패 | API/mapper layer |
| UI Error | 렌더링 중 예외 | Error Boundary |
| Unknown Error | 예상 못한 오류 | fallback handler, reporter |

## 책임 위치

| 책임 | 위치 |
| --- | --- |
| HTTP 요청 실패 감지 | `apiClient`, API function |
| API 응답 구조 검증 | `schema`, mapper |
| 비즈니스 실패 분류 | use-case hook |
| 사용자 alert/toast/dialog | use-case hook 또는 page boundary |
| 입력값 오류 표시 | form component |
| 렌더링 오류 복구 | Error Boundary |
| 로깅/모니터링 | 공통 error reporter |

## 기본 원칙

1. API 함수는 사용자 피드백을 직접 띄우지 않는다.
2. 컴포넌트마다 `try/catch`를 반복하지 않는다.
3. 에러를 삼키지 않는다.
4. 사용자 메시지와 개발자 로그를 분리한다.
5. 서버 응답은 schema로 검증하고 deserialize 실패를 명시적으로 처리한다.
6. optimistic update에는 rollback 전략이 있어야 한다.
7. 렌더링 오류와 API business error를 같은 방식으로 처리하지 않는다.

## 1. API 함수에서 alert 하지 않는다

API 함수는 순수 HTTP 요청과 응답 반환에 집중한다.

### Do

```ts
export const cancelOrder = async (orderId: number) => {
  return apiClient.post("/cancel", { orderId });
};
```

```ts
const useCancelOrder = () => {
  const queryClient = useQueryClient();
  const { alert } = useDialog();

  return useMutation({
    mutationFn: cancelOrder,
    onSuccess: () => {
      alert({ title: "완료", content: "주문이 취소되었습니다." });
      queryClient.invalidateQueries({ queryKey: ["orders"] });
    },
    onError: (error) => {
      alert({
        title: "ERROR",
        content: getCancelOrderErrorMessage(error),
      });
    },
  });
};
```

### Do Not

```ts
export const cancelOrder = async (orderId: number) => {
  try {
    return await apiClient.post("/cancel", { orderId });
  } catch (error) {
    alert("주문 취소 실패");
    throw error;
  }
};
```

API 함수가 alert을 띄우면 같은 API를 다른 화면에서 다른 방식으로 처리하기 어렵다.

## 2. 컴포넌트에서 try/catch를 남발하지 않는다

컴포넌트는 에러 처리 정책을 직접 소유하지 않는다. 사용자 액션의 성공/실패 처리는 use-case hook에서 관리한다.

### Do

```ts
const useCancelOrder = () => {
  const queryClient = useQueryClient();
  const { alert } = useDialog();

  return useMutation({
    mutationFn: cancelOrder,
    onSuccess: () => {
      alert({ title: "완료", content: "주문이 취소되었습니다." });
      queryClient.invalidateQueries({ queryKey: ["orders"] });
    },
    onError: (error) => {
      alert({
        title: "ERROR",
        content: getCancelOrderErrorMessage(error),
      });
    },
  });
};
```

```tsx
const OrderCancelButton = ({
  disabled,
  onClick,
}: {
  disabled: boolean;
  onClick: () => void;
}) => {
  return (
    <Button disabled={disabled} onClick={onClick}>
      주문 취소
    </Button>
  );
};
```

### Do Not

```tsx
const OrderCancelButton = ({ orderId }: { orderId: number }) => {
  const { alert } = useDialog();

  const handleClick = async () => {
    try {
      await cancelOrder(orderId);
      alert({ title: "완료", content: "주문이 취소되었습니다." });
    } catch {
      alert({ title: "ERROR", content: "주문 취소에 실패했습니다." });
    }
  };

  return <Button onClick={handleClick}>주문 취소</Button>;
};
```

컴포넌트에 API 호출과 에러 처리 정책이 섞이면 재사용과 테스트가 어려워진다.

## 3. 사용자 메시지와 개발자 로그를 분리한다

사용자 메시지는 짧고 조치 가능해야 한다.

개발자 로그는 원인 분석에 필요한 기술 정보를 포함한다.

### Do

```ts
const getCancelOrderErrorMessage = (error: unknown) => {
  if (isPermissionError(error)) {
    return "주문을 취소할 권한이 없습니다.";
  }

  if (isAlreadyShippedError(error)) {
    return "이미 출고된 주문은 취소할 수 없습니다.";
  }

  return "주문 취소에 실패했습니다. 잠시 후 다시 시도해주세요.";
};
```

```ts
const handleCancelOrderError = (error: unknown) => {
  reportError(error, {
    feature: "outbound-order",
    action: "cancel-order",
  });

  return getCancelOrderErrorMessage(error);
};
```

### Do Not

```ts
const getCancelOrderErrorMessage = (error: unknown) => {
  return String((error as Error).message);
};
```

서버 에러 메시지나 stack trace를 그대로 사용자에게 노출하지 않는다.

## 4. 에러를 삼키지 않는다

에러를 잡고 아무것도 하지 않으면 호출자는 실패를 알 수 없다.

### Do

```ts
const runCancelOrder = async (orderId: number) => {
  try {
    await cancelOrder(orderId);
  } catch (error) {
    reportError(error, {
      feature: "outbound-order",
      action: "cancel-order",
    });

    throw error;
  }
};
```

### Do Not

```ts
const runCancelOrder = async (orderId: number) => {
  try {
    await cancelOrder(orderId);
  } catch (error) {
    console.error(error);
  }
};
```

`catch` 후 다시 throw하지 않으면 상위 hook이나 UI는 실패 여부를 알 수 없다.

## 5. Zod deserialize 실패를 명시적으로 처리한다

서버 응답 구조가 기대와 다르면 런타임에서 조용히 깨지지 않도록 schema 검증 결과를 명시적으로 반환한다.

### Do

```ts
type FetchResult<T> =
  | { success: true; data: ApiResponse<T> }
  | { success: false; reason: "DESERIALIZE_FAILED"; error: z.ZodError }
  | { success: false; reason: "REQUEST_FAILED"; error: unknown };

export const fetchOrderList = async (
  params: OrderListRequest,
): Promise<FetchResult<OrderListResponseData>> => {
  let response: ApiResponse<unknown>;

  try {
    response = await apiClient.post("/orders", params);
  } catch (error) {
    return { success: false, reason: "REQUEST_FAILED", error };
  }

  const parsed = orderListResponseDataSchema.safeParse(response.data);

  if (!parsed.success) {
    return {
      success: false,
      reason: "DESERIALIZE_FAILED",
      error: parsed.error,
    };
  }

  return {
    success: true,
    data: { ...response, data: parsed.data },
  };
};
```

```ts
const useOrderList = (params: OrderListRequest) => {
  return useQuery({
    queryKey: ["orders", params],
    queryFn: () => fetchOrderList(params),
    select: (result) => {
      if (!result.success) {
        throw result;
      }

      return result.data.data;
    },
  });
};
```

### Do Not

```ts
export const fetchOrderList = async (params: OrderListRequest) => {
  const response = await apiClient.post("/orders", params);

  return response.data as OrderListResponseData;
};
```

`as` 캐스팅으로 서버 응답 구조 불일치를 숨기지 않는다.

## 6. Optimistic update 실패에는 rollback이 있어야 한다

서버 응답 전에 UI를 먼저 바꾸는 경우 실패 시 이전 상태로 되돌릴 수 있어야 한다.

### Do

```ts
const useToggleFavoriteOrder = () => {
  const queryClient = useQueryClient();

  return useMutation({
    mutationFn: toggleFavoriteOrder,

    onMutate: async ({ orderId }) => {
      await queryClient.cancelQueries({ queryKey: ["orders"] });

      const previousOrders = queryClient.getQueryData<OrderListResponse>(["orders"]);

      queryClient.setQueryData<OrderListResponse>(["orders"], (current) => {
        if (!current) {
          return current;
        }

        return {
          ...current,
          orders: current.orders.map((order) =>
            order.id === orderId ? { ...order, favorite: !order.favorite } : order,
          ),
        };
      });

      return { previousOrders };
    },

    onError: (_error, _variables, context) => {
      if (context?.previousOrders) {
        queryClient.setQueryData(["orders"], context.previousOrders);
      }
    },

    onSettled: () => {
      queryClient.invalidateQueries({ queryKey: ["orders"] });
    },
  });
};
```

### Do Not

```ts
const toggleFavorite = async (orderId: number) => {
  setOrders((prev) =>
    prev.map((order) =>
      order.id === orderId ? { ...order, favorite: !order.favorite } : order,
    ),
  );

  await toggleFavoriteOrder({ orderId });
};
```

실패 시 rollback이 없다면 optimistic update를 사용하지 않는다.

## 7. Error Boundary는 렌더링 실패를 복구한다

Error Boundary는 렌더링 중 발생한 예외를 잡고 fallback UI를 보여주기 위한 장치다.

API business error를 처리하는 곳이 아니다.

### Do

```tsx
<ErrorBoundary fallback={<PageErrorFallback />}>
  <OutboundListPage />
</ErrorBoundary>
```

### Do Not

```tsx
<ErrorBoundary fallback={<OrderCancelFailedFallback />}>
  <OrderCancelButton />
</ErrorBoundary>
```

주문 취소 실패 같은 사용자 액션 에러는 mutation hook의 `onError`에서 처리한다.

## 8. 에러 피드백 기준

| 상황 | 피드백 |
| --- | --- |
| 사용자가 실행한 명령 실패 | dialog 또는 toast |
| 폼 입력 오류 | field error |
| 백그라운드 refetch 실패 | 조용한 retry 또는 작은 toast |
| 권한 없음 | 접근 불가 안내 또는 redirect |
| 로그인 만료 | 로그인 이동 |
| 렌더링 실패 | fallback UI |
| 개발자만 알아야 하는 오류 | logging/reporting |

## 금지 패턴 요약

### API 함수 안에서 alert 띄우기

```ts
catch (error) {
  alert("실패했습니다.");
  throw error;
}
```

사용자 피드백은 use-case layer에서 결정한다.

### 컴포넌트마다 try/catch 반복

```tsx
try {
  await cancelOrder(orderId);
} catch {
  alert("실패했습니다.");
}
```

사용자 액션의 성공/실패 정책은 hook으로 분리한다.

### 에러 삼키기

```ts
catch (error) {
  console.error(error);
}
```

상위 호출자가 실패를 알 수 없게 된다.

### 서버 에러 메시지 그대로 노출

```ts
alert({ content: String(error.message) });
```

사용자 메시지는 조치 가능한 문장으로 변환한다.

### Zod parse 실패 무시

```ts
return response.data as OrderListResponseData;
```

서버 응답 계약 불일치를 숨기지 않는다.

### Error Boundary로 business error 처리

```tsx
<ErrorBoundary>
  <OrderCancelButton />
</ErrorBoundary>
```

렌더링 실패와 사용자 액션 실패는 처리 계층이 다르다.

## 판단 기준

```md
1. 이 에러는 어디서 감지되는가?
   -> API, schema, form, render, user action 중 하나로 분류한다.

2. 이 에러의 의미를 가장 잘 아는 계층은 어디인가?
   -> use-case hook 또는 domain helper에서 해석한다.

3. 사용자가 취할 수 있는 행동이 있는가?
   -> 있다면 dialog/toast/field error로 안내한다.

4. 개발자만 알아야 하는 정보인가?
   -> error reporter로 보낸다.

5. 실패 후 복구가 필요한가?
   -> rollback, retry, invalidate 전략을 명시한다.

6. 렌더링 중 예외인가?
   -> Error Boundary에서 fallback UI를 제공한다.
```

## 결론

좋은 에러 핸들링은 모든 곳에 `try/catch`를 넣는 것이 아니다.

API layer는 실패를 감지하고, schema/mapper는 데이터 계약을 검증하고, use-case hook은 실패 의미와 사용자 피드백을 결정하고, UI는 사용자가 이해할 수 있는 결과만 표현한다.
