# 6. 프론트엔드 에러 모델링

## 목적

도메인적으로 의미 있는 실패를 명확한 에러 모델로 표현하고, UI 피드백과 비즈니스 로직을 분리한다.

에러 모델링은 모든 실패를 `catch`로 뭉개는 것이 아니라, 도메인적으로 의미 있는 실패를 타입과 이름으로 드러내는 것이다.

## 핵심 문장

비즈니스 로직은 실패의 의미를 `throw`하고, UI 피드백은 use-case layer의 데코레이터나 mutation callback에서 처리한다.

도메인 로직 안에서 alert, toast, dialog를 직접 호출하지 않는다. 도메인 로직은 "왜 실패했는지"를 알려주고, 사용자에게 "어떻게 보여줄지"는 바깥 계층이 결정한다.

## 적용 범위

에러 클래스와 고차 함수 기반 에러 처리는 모든 코드에 적용하는 기본값이 아니다.

다음처럼 도메인 실패 의미가 중요하고, 반복되는 사용자 액션 처리 패턴이 있을 때 적용한다.

- 바코드 스캔
- 포장 수량 변경
- 박스 변경
- 송장 출력
- 출고 확정
- 작업자 할당
- 대량 처리
- 재고 검증

단순 조회 실패, 단순 form field validation, 일반적인 React Query 에러 처리에는 과한 구조일 수 있다.

## 기본 구조

```md
1. domain helper는 비즈니스 조건을 검사하고 의미 있는 domain error를 throw한다.
2. API 함수는 HTTP 요청만 한다.
3. use-case hook은 domain validation + mutation/query + invalidate를 조립한다.
4. UI action layer는 고차 함수/decorator로 성공/실패 피드백을 붙인다.
5. component는 action을 호출할 뿐 error class나 alert 정책을 직접 알지 않는다.
```

## 1. 도메인 에러 클래스로 실패 의미를 표현한다

도메인 에러 클래스는 "기술적으로 실패했다"가 아니라 "업무적으로 왜 실패했는지"를 표현한다.

### Do

```ts
class PackingError extends Error {
  constructor(message: string) {
    super(message);
    this.name = "PackingError";
  }
}

class InvalidBarcodeError extends PackingError {
  constructor(barcode: string) {
    super(`유효하지 않은 바코드입니다: ${barcode}`);
    this.name = "InvalidBarcodeError";
  }
}

class QuantityExceededError extends PackingError {
  constructor(itemName: string, maxQty: number, requestedQty: number) {
    super(`${itemName}은 ${maxQty}개까지만 처리할 수 있습니다. 요청 수량: ${requestedQty}개`);
    this.name = "QuantityExceededError";
  }
}

class AlreadyPackedError extends PackingError {
  constructor() {
    super("이미 포장 완료된 상품입니다.");
    this.name = "AlreadyPackedError";
  }
}
```

### Do Not

```ts
throw new Error("오류가 발생했습니다.");
```

```ts
throw new Error("BAD_REQUEST");
```

에러 이름과 메시지가 도메인 실패 의미를 설명하지 못하면 호출자는 실패를 구분할 수 없다.

## 2. 비즈니스 로직은 UI 알림을 직접 호출하지 않는다

도메인 로직은 상태 변경 가능 여부와 실패 조건만 판단한다.

### Do

```ts
const validatePackingScan = ({
  barcode,
  item,
  requestedQty,
}: {
  barcode: string;
  item: PackingItem;
  requestedQty: number;
}) => {
  if (!barcode) {
    throw new InvalidBarcodeError(barcode);
  }

  if (item.status === "PACKED") {
    throw new AlreadyPackedError();
  }

  if (requestedQty > item.remainingQty) {
    throw new QuantityExceededError(item.name, item.remainingQty, requestedQty);
  }
};
```

### Do Not

```ts
const validatePackingScan = ({
  barcode,
  item,
  requestedQty,
  alert,
}: {
  barcode: string;
  item: PackingItem;
  requestedQty: number;
  alert: (message: string) => void;
}) => {
  if (!barcode) {
    alert("유효하지 않은 바코드입니다.");
    return false;
  }

  if (requestedQty > item.remainingQty) {
    alert("수량이 재고를 초과했습니다.");
    return false;
  }

  return true;
};
```

비즈니스 로직이 alert에 의존하면 알림 방식이 바뀔 때 도메인 로직을 수정해야 한다.

## 3. API 함수는 HTTP 요청만 한다

에러 모델링을 하더라도 API 함수에 UI 피드백을 넣지 않는다.

### Do

```ts
const scanPackingItem = async (payload: ScanPackingItemRequest) => {
  return apiClient.post("/packing/scan", payload);
};
```

### Do Not

```ts
const scanPackingItem = async (payload: ScanPackingItemRequest) => {
  try {
    return await apiClient.post("/packing/scan", payload);
  } catch (error) {
    alert("스캔 처리에 실패했습니다.");
    throw error;
  }
};
```

사용자 피드백은 API layer가 아니라 use-case layer에서 결정한다.

## 4. use-case hook에서 domain validation과 mutation을 조립한다

use-case hook은 도메인 검증, 서버 요청, 캐시 갱신을 연결한다.

### Do

```ts
const useScanPackingItem = () => {
  const queryClient = useQueryClient();

  return useMutation({
    mutationFn: async (command: ScanPackingCommand) => {
      validatePackingScan(command);

      return scanPackingItem({
        barcode: command.barcode,
        qty: command.requestedQty,
      });
    },
    onSuccess: () => {
      queryClient.invalidateQueries({ queryKey: ["packing-items"] });
    },
  });
};
```

### Do Not

```ts
const useScanPackingItem = () => {
  const queryClient = useQueryClient();
  const { alert } = useDialog();

  return useMutation({
    mutationFn: async (command: ScanPackingCommand) => {
      if (command.requestedQty > command.item.remainingQty) {
        alert({ title: "ERROR", content: "수량이 재고를 초과했습니다." });
        return;
      }

      return scanPackingItem({
        barcode: command.barcode,
        qty: command.requestedQty,
      });
    },
    onSuccess: () => {
      queryClient.invalidateQueries({ queryKey: ["packing-items"] });
    },
  });
};
```

mutation 내부에서 도메인 조건과 UI alert을 함께 처리하면 실패 의미와 피드백 방식이 결합된다.

## 5. 고차 함수로 UI 피드백 책임을 분리한다

반복되는 성공/실패 알림 처리는 고차 함수로 감쌀 수 있다.

이 구조는 함수형 스타일의 decorator 패턴이다. 기존 action의 핵심 동작은 유지하고, 바깥에서 성공/실패 알림이라는 부가 동작을 붙인다.

### Do

```ts
const getPackingErrorMessage = (error: unknown) => {
  if (error instanceof PackingError) {
    return error.message;
  }

  return "포장 처리 중 오류가 발생했습니다.";
};
```

```ts
export const withTryNotifySuccess = <Args extends readonly unknown[], Result>(
  action: (...args: Args) => Promise<Result>,
  successMessage: string,
  addNotification: (message: string, type: "success" | "error") => void,
  getErrorMessage: (error: unknown) => string = getPackingErrorMessage,
) => {
  return async (...args: Args): Promise<Result | undefined> => {
    try {
      const result = await action(...args);
      addNotification(successMessage, "success");
      return result;
    } catch (error) {
      addNotification(getErrorMessage(error), "error");
      return undefined;
    }
  };
};
```

```ts
const scanPackingItem = useScanPackingItem();
const { addNotification } = useNotification();

const handleScan = withTryNotifySuccess(
  async (command: ScanPackingCommand) => {
    return scanPackingItem.mutateAsync(command);
  },
  "스캔 처리되었습니다.",
  addNotification,
  getPackingErrorMessage,
);
```

alert dialog를 사용하는 화면이라면 같은 패턴을 alert에 맞춘 고차 함수로 만들 수 있다.

```ts
export const withTryAlertSuccess = <Args extends readonly unknown[], Result>(
  action: (...args: Args) => Promise<Result>,
  successMessage: string,
  alert: (params: { title: string; content: string }) => void,
  getErrorMessage: (error: unknown) => string = getPackingErrorMessage,
) => {
  return async (...args: Args): Promise<Result | undefined> => {
    try {
      const result = await action(...args);
      alert({ title: "완료", content: successMessage });
      return result;
    } catch (error) {
      alert({ title: "ERROR", content: getErrorMessage(error) });
      return undefined;
    }
  };
};
```

```ts
const scanPackingItem = useScanPackingItem();
const { alert } = useDialog();

const handleScan = withTryAlertSuccess(
  async (command: ScanPackingCommand) => {
    return scanPackingItem.mutateAsync(command);
  },
  "스캔 처리되었습니다.",
  alert,
  getPackingErrorMessage,
);
```

컴포넌트는 `handleScan`을 호출할 뿐 에러 클래스나 alert/notification 정책을 직접 알지 않는다.

### Do Not

```tsx
const PackingScanButton = ({ command }: { command: ScanPackingCommand }) => {
  const scanPackingItem = useScanPackingItem();
  const { alert } = useDialog();

  const handleScan = async () => {
    try {
      await scanPackingItem.mutateAsync(command);
      alert({ title: "완료", content: "스캔 처리되었습니다." });
    } catch (error) {
      if (error instanceof PackingError) {
        alert({ title: "ERROR", content: error.message });
        return;
      }

      alert({ title: "ERROR", content: "포장 처리 중 오류가 발생했습니다." });
    }
  };

  return <Button onClick={handleScan}>스캔</Button>;
};
```

컴포넌트마다 try/catch와 에러 메시지 분기 처리가 반복된다.

## 6. `return undefined`는 조심해서 사용한다

고차 함수에서 에러를 잡고 `undefined`를 반환하면 호출자는 실패를 모를 수 있다.

이벤트 핸들러처럼 결과를 사용하지 않는 곳에서는 허용할 수 있다. 하지만 호출자가 결과를 반드시 기대하는 경우에는 실패를 다시 throw하거나 Result 타입을 반환한다.

### 이벤트 핸들러에서는 허용 가능

```ts
const handleScan = withTryNotifySuccess(
  async (command: ScanPackingCommand) => {
    await scanPackingItem.mutateAsync(command);
  },
  "스캔 처리되었습니다.",
  addNotification,
  getPackingErrorMessage,
);
```

### 결과가 필요한 경우에는 throw 유지

```ts
const withTryNotifySuccessAndRethrow = <Args extends readonly unknown[], Result>(
  action: (...args: Args) => Promise<Result>,
  successMessage: string,
  addNotification: (message: string, type: "success" | "error") => void,
  getErrorMessage: (error: unknown) => string = getPackingErrorMessage,
) => {
  return async (...args: Args): Promise<Result> => {
    try {
      const result = await action(...args);
      addNotification(successMessage, "success");
      return result;
    } catch (error) {
      addNotification(getErrorMessage(error), "error");
      throw error;
    }
  };
};
```

## 7. 모든 에러를 class로 만들지 않는다

에러 클래스는 도메인 의미가 있는 실패에만 사용한다.

### 적합한 예

```md
- QuantityExceededError
- InvalidBarcodeError
- AlreadyPackedError
- MissingWorkerError
- PrinterUnavailableError
- EmptyCartError
```

### 과한 예

```md
- UnknownApiError
- CommonError
- BadRequestError를 화면마다 새로 class화
- 단순 form required error를 class화
```

도메인 의미가 없는 기술적 실패는 공통 error parser나 API client 레벨에서 처리한다.

## WMS 적용 후보

| 유스케이스 | 모델링할 수 있는 에러 |
| --- | --- |
| 바코드 스캔 | `InvalidBarcodeError`, `UnknownBarcodeError`, `AlreadyScannedError` |
| 포장 수량 변경 | `QuantityExceededError`, `InvalidQuantityError` |
| 박스 변경 | `InvalidBoxError`, `BoxChangeNotAllowedError` |
| 송장 출력 | `PrinterUnavailableError`, `InvoiceNotIssuedError` |
| 출고 확정 | `OutboundNotReadyError`, `PackingIncompleteError` |
| 작업자 할당 | `MissingWorkerError`, `WorkerAssignmentNotAllowedError` |
| 대량 처리 | `PartialFailureError`, `BatchLimitExceededError` |

## 금지 패턴 요약

### 도메인 로직에서 alert 호출

```ts
if (remainingQty <= 0) {
  alert("재고가 부족합니다.");
  return;
}
```

도메인 로직은 실패 의미를 throw하고, UI 피드백은 바깥에서 처리한다.

### API 함수에 데코레이터 적용

```ts
export const scanPackingItem = withTryNotifySuccess(apiCall, "성공했습니다.", addNotification);
```

API 함수는 HTTP 요청만 담당한다. 데코레이터는 use-case action layer에서 적용한다.

### 모든 실패를 문자열로 처리

```ts
throw new Error("실패했습니다.");
```

도메인 실패는 이름과 타입으로 구분할 수 있어야 한다.

### 컴포넌트마다 try/catch 반복

```tsx
try {
  await action();
} catch (error) {
  alert({ content: String(error) });
}
```

반복되는 성공/실패 피드백은 고차 함수나 use-case hook으로 분리한다.

## 판단 기준

```md
1. 이 실패가 도메인적으로 의미 있는가?
   -> 의미가 있다면 domain error class를 고려한다.

2. 이 실패를 여러 UI에서 다르게 보여줄 수 있는가?
   -> 비즈니스 로직에서 alert하지 말고 throw한다.

3. 성공/실패 피드백 패턴이 반복되는가?
   -> 고차 함수/decorator로 분리한다.

4. action 결과를 호출자가 사용해야 하는가?
   -> 에러를 삼키지 말고 throw 또는 Result 타입을 고려한다.

5. 단순 기술적 실패인가?
   -> API client, error parser, React Query onError에서 처리한다.

6. 에러 클래스가 도메인 용어를 드러내는가?
   -> 아니면 class로 만들 필요가 약하다.
```

## 결론

에러 클래스와 고차 함수는 비즈니스 로직의 순수성을 지키고 UI 피드백을 일관되게 처리하기 위한 도구다.

다만 모든 실패를 class로 만들거나 모든 action을 데코레이터로 감싸는 것은 과하다.

WMS에서는 바코드 스캔, 포장 수량 변경, 송장 출력, 출고 확정처럼 도메인 실패 의미가 중요하고 반복되는 사용자 액션에 제한적으로 적용한다.
