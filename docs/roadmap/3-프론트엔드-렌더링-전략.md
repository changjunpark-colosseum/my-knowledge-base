# 3. 프론트엔드 렌더링 전략

## 목적

화면의 데이터 성격에 따라 CSR, SSR, SSG, ISR을 올바르게 선택한다.

렌더링 전략은 성능 최적화 옵션이 아니라 데이터의 최신성, 사용자 개인화, SEO 필요성, 캐시 가능성에 따라 선택하는 화면 설계 결정이다.

## 핵심 문장

CGKR의 기본 렌더링 전략은 CSR이다.

CGKR의 핵심 화면은 로그인 기반 업무 화면이며, 사용자별 권한, 센터, 화주, 검색 조건에 따라 데이터가 달라진다. 또한 테이블 조작, 스캔, 출력, mutation 후 갱신처럼 클라이언트 상호작용이 많다.

따라서 기본 전략은 CSR + React Query로 두고, SSR은 공개 페이지, SEO가 필요한 페이지, 요청 시점 HTML이 반드시 필요한 페이지에서만 선택한다.

## 렌더링 전략 요약

| 전략 | 실행 시점 | 적합한 화면 |
| --- | --- | --- |
| CSR | 브라우저에서 렌더링 | 로그인 후 업무 화면, 사용자별 데이터 |
| SSR | 요청마다 서버에서 렌더링 | 최신성이 중요하고 SEO가 필요한 화면 |
| SSG | 빌드 시점에 정적 생성 | 거의 변하지 않는 공개 페이지 |
| ISR | 빌드 후 주기적으로 정적 재생성 | 자주 바뀌지만 실시간은 아닌 공개 페이지 |

## 선택 기준

| 질문 | 선택 |
| --- | --- |
| 로그인 사용자별 데이터인가? | CSR |
| SEO가 중요한 공개 페이지인가? | SSR, SSG, ISR |
| 요청마다 최신 데이터가 필요한가? | SSR |
| 빌드 시점 데이터로 충분한가? | SSG |
| 일정 주기로 갱신되어도 괜찮은가? | ISR |
| 브라우저 API나 사용자 입력 중심인가? | CSR |
| 서버에서 미리 그리면 민감 정보 노출 위험이 있는가? | CSR 우선 |

## CGKR에서 SSR을 기본값으로 두지 않는 이유

SSR은 금지가 아니라 예외 전략이다. 다만 CGKR 업무 화면에서 SSR을 기본으로 잡으면 이점보다 비용과 리스크가 크다.

### 1. 대부분 로그인 후 사용자별 데이터다

출고, 입고, 재고, 포장, 관리자 화면은 사용자, 센터, 권한, 화주, 메뉴 권한에 따라 데이터가 달라진다.

SSR로 가져오려면 서버 렌더링 시점에 쿠키, 세션, 권한, 센터 컨텍스트를 모두 정확히 해석해야 한다.

### 2. 캐시 이점이 거의 없다

SSR은 요청마다 서버에서 HTML을 만든다.

CGKR 화면은 사용자별 결과가 달라서 CDN이나 공용 캐시를 적극적으로 쓰기 어렵다. 결국 매 요청마다 서버 비용은 늘지만 정적 캐시 이점은 작다.

### 3. 상호작용 중심 화면이 많다

검색, 필터, 페이지네이션, 테이블 선택, 모달, 스캔, 출력, mutation 후 invalidate 같은 흐름이 많다.

어차피 브라우저에서 React Query로 계속 데이터를 갱신해야 하므로 SSR로 첫 데이터만 내려도 이후 상태 관리는 CSR 쪽으로 다시 넘어간다.

### 4. 인증/권한 경계가 복잡해진다

SSR에서 서버가 사용자별 데이터를 props로 내려주면 캐시 헤더 실수나 shared response 처리 실수 때 민감 데이터 노출 리스크가 생긴다.

내부 업무 시스템에서는 이 리스크를 감수할 만큼 SEO 이점이 거의 없다.

### 5. Next.js 서버 부하가 증가한다

CSR은 정적 JavaScript를 받고 브라우저가 API를 호출한다.

SSR은 페이지 요청마다 Next 서버가 데이터 조회와 렌더링을 수행한다. 테이블 중심 업무 화면에서 사용자가 많아지면 Next 서버가 API proxy + render worker처럼 부담을 떠안게 된다.

### 6. SEO 필요가 없다

CGKR의 핵심 화면은 로그인 뒤 접근하는 내부 업무 화면이다.

검색엔진 노출이 필요 없으므로 SSR의 대표 장점인 SEO 개선이 거의 의미 없다.

## 1. CSR

Client Side Rendering.

브라우저에서 JavaScript를 실행한 뒤 데이터를 가져와 화면을 렌더링한다.

### 적합한 경우

- 로그인 후 내부 업무 시스템
- 사용자 권한에 따라 달라지는 화면
- 검색, 필터, 테이블 조작이 많은 화면
- SEO가 중요하지 않은 화면
- 브라우저 상태와 상호작용이 많은 화면
- mutation 후 invalidate가 많은 화면

### CGKR 예시

- 출고 목록
- 입고 목록
- 재고 관리
- 포장/스캔 화면
- 대시보드
- 관리자 설정

### Do

```tsx
const OutboundListPage = () => {
  const { params, setSearchParams } = useOutboundSearchParams();
  const { data, isLoading } = useOutboundList(params);

  return (
    <>
      <OutboundSearchForm
        defaultValues={params}
        onSubmit={(values) => setSearchParams({ ...values, page: 1 })}
      />
      <OutboundTable rows={data?.items ?? []} loading={isLoading} />
    </>
  );
};
```

### Do Not

```tsx
export const getServerSideProps = async (context) => {
  const token = context.req.cookies.token;
  const outboundList = await fetchOutboundList({ token });

  return {
    props: {
      outboundList,
    },
  };
};
```

SEO가 필요 없는 업무 테이블 화면을 SSR로 만들지 않는다.

이 방식은 요청마다 Next 서버에서 인증 컨텍스트 해석, API 호출, HTML 렌더링을 수행하게 만든다. 하지만 사용자는 이후 검색, 필터, 페이지네이션, mutation에서 다시 클라이언트 데이터 조회를 수행하므로 SSR 이점이 작다.

## 2. SSR

Server Side Rendering.

요청이 들어올 때마다 서버에서 HTML을 생성한다.

### 적합한 경우

- SEO가 중요하다.
- 요청 시점의 최신 데이터가 필요하다.
- 공개 페이지지만 사용자 요청별로 내용이 달라진다.
- 메타 태그가 데이터에 따라 달라진다.
- 서버에서 HTML을 만들어야 하는 명확한 이유가 있다.

### Do

```tsx
export const getServerSideProps = async ({ params }) => {
  const product = await fetchPublicProductDetail(params.productId);

  return {
    props: {
      product,
    },
  };
};
```

### Do Not

```tsx
export const getServerSideProps = async () => {
  const dashboard = await fetchPrivateDashboard();

  return {
    props: {
      dashboard,
    },
  };
};
```

SEO가 필요 없는 내부 업무 대시보드를 SSR로 만들면 요청마다 서버 렌더링 비용만 늘어난다.

## 3. SSG

Static Site Generation.

빌드 시점에 HTML을 미리 생성한다.

### 적합한 경우

- 내용이 거의 변하지 않는다.
- 모든 사용자에게 같은 내용을 보여준다.
- SEO가 중요하다.
- 빠른 초기 응답이 중요하다.
- 빌드 시점 데이터로 충분하다.

### 예시

- 약관
- 도움말
- 공개 문서
- 소개 페이지
- 정적 랜딩 페이지

### Do

```tsx
export const getStaticProps = async () => {
  const guide = await fetchGuideContent();

  return {
    props: {
      guide,
    },
  };
};
```

### Do Not

```tsx
export const getStaticProps = async () => {
  const myOrders = await fetchMyOrders();

  return {
    props: {
      myOrders,
    },
  };
};
```

사용자별 데이터는 빌드 시점에 정적으로 만들 수 없다.

## 4. ISR

Incremental Static Regeneration.

정적 페이지를 생성하되, 일정 주기마다 백그라운드에서 다시 생성한다.

### 적합한 경우

- 공개 페이지다.
- SEO가 중요하다.
- 데이터가 바뀌지만 실시간 최신성은 필요 없다.
- 캐시된 페이지를 보여줘도 괜찮다.
- 정적 응답 속도와 주기적 데이터 갱신이 모두 필요하다.

### 예시

- 상품 상세
- 공지사항 상세
- 블로그 글
- 공개 재고 요약
- 이벤트 페이지

### Do

```tsx
export const getStaticProps = async ({ params }) => {
  const product = await fetchPublicProductDetail(params.productId);

  return {
    props: {
      product,
    },
    revalidate: 60,
  };
};
```

### Do Not

```tsx
export const getStaticProps = async () => {
  const currentUser = await fetchCurrentUser();

  return {
    props: {
      currentUser,
    },
    revalidate: 60,
  };
};
```

ISR은 사용자별 개인 데이터를 다루는 전략이 아니다.

## CGKR 기준

CGKR 같은 로그인 기반 업무 시스템은 대부분 CSR을 기본값으로 둔다.

| 화면 | 기본 전략 | 이유 |
| --- | --- | --- |
| 출고 목록 | CSR | 검색, 필터, 테이블 조작, 사용자별 권한 |
| 입고 목록 | CSR | 사용자별 센터/화주 데이터 |
| 재고 관리 | CSR | 최신 서버 상태, mutation 후 갱신 |
| 포장/스캔 화면 | CSR | 브라우저 상호작용, 실시간 입력 |
| 대시보드 | CSR | 로그인 사용자별 데이터 |
| 관리자 설정 | CSR | 권한 기반 화면 |
| 공개 문서/도움말 | SSG 또는 ISR | 캐시 가능, 사용자별 데이터 아님 |
| SEO가 필요한 공개 페이지 | SSR 또는 ISR | 검색엔진 노출과 메타 정보 필요 |

## 선택 기준 문장

```md
1. 이 페이지가 로그인 후 내부 업무 화면인가?
   -> CSR을 기본으로 한다.

2. SEO가 중요한 공개 페이지인가?
   -> SSR, SSG, ISR 중 선택한다.

3. 데이터가 요청마다 반드시 최신이어야 하는가?
   -> SSR을 고려한다.

4. 데이터가 거의 변하지 않는가?
   -> SSG를 고려한다.

5. 정적 캐시는 가능하지만 주기적 갱신이 필요한가?
   -> ISR을 고려한다.

6. 사용자별 민감 데이터인가?
   -> CSR을 우선 고려하고, 서버 렌더링 시 캐시/인증 경계를 명확히 한다.
```

## 금지 패턴 요약

### SEO가 필요 없는 업무 화면을 SSR로 만들기

```tsx
export const getServerSideProps = async (context) => {
  const orders = await fetchOrdersByUser(context.req.cookies.token);

  return {
    props: {
      orders,
    },
  };
};
```

사용자별 업무 데이터는 캐시 이점이 작고 서버 렌더링 비용과 인증 복잡도만 커질 수 있다.

### 사용자별 데이터를 SSG/ISR로 만들기

```tsx
export const getStaticProps = async () => {
  const currentUser = await fetchCurrentUser();

  return {
    props: {
      currentUser,
    },
    revalidate: 60,
  };
};
```

SSG와 ISR은 사용자별 민감 데이터에 맞지 않는다.

### 렌더링 전략을 성능 감으로만 선택하기

```md
SSR이면 빠르다.
SSG면 무조건 좋다.
CSR은 느리다.
```

렌더링 전략은 데이터 소유자, 최신성, 캐시 가능성, SEO 필요성, 사용자 개인화 여부로 선택한다.

## 결론

CGKR에서 SSR을 쓰지 말자는 것이 아니다.

SSR을 "성능이 좋아 보이는 옵션"으로 기본 적용하지 말자는 것이다. CGKR 업무 화면의 기본값은 CSR이고, SSR은 SEO, 공개 페이지, 요청 시점 HTML 필요성이 명확할 때만 선택한다.
