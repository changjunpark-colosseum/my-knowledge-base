# 4. 프론트엔드 추상화 레벨

## 목적

프론트엔드 코드에서 어떤 로직을 어디까지 추상화할지 판단 기준을 세운다.

추상화는 중복 제거가 아니라 변경 이유를 분리하는 수단이다. 잘못된 추상화는 코드 양은 줄이지만, 서로 다른 유스케이스를 강제로 묶어서 변경을 어렵게 만든다.

참고: [추상화를 정렬하기](https://junilhwang.github.io/TIL/clean-code/%EC%A1%B0%EA%B0%81%EB%AA%A8%EC%9D%8C/%EC%B6%94%EC%83%81%ED%99%94%EB%A5%BC-%EC%A0%95%EB%A0%AC%ED%95%98%EA%B8%B0/)

## 핵심 문장

추출과 추상화는 다르다.

추출은 코드를 다른 위치로 옮기는 것이고, 추상화는 변경 이유와 사용 의도를 기준으로 새로운 경계를 만드는 것이다.

함수를 내부로 빼거나 파일을 나누었다고 해서 자동으로 추상화가 되지 않는다. 이름, 입력, 출력, 책임이 명확하지 않은 추출은 단지 코드를 숨긴 것일 뿐이다.

## 추출과 추상화는 다르다

### 추출

추출은 코드 조각을 함수, hook, 파일 등으로 옮기는 행위다.

```tsx
const OrderCancelButton = ({ order }: { order: Order }) => {
  const disabled = getDisabled(order);

  return <Button disabled={disabled}>취소</Button>;
};

const getDisabled = (order: Order) => {
  return order.status !== "READY" || order.invoiceIssued;
};
```

이 코드는 조건식을 함수로 추출했지만 좋은 추상화라고 보기 어렵다.

`getDisabled`는 무엇의 disabled인지, 어떤 도메인 판단인지 드러내지 않는다. 구현은 숨겼지만 의미는 드러나지 않았다.

### 추상화

추상화는 코드의 의도와 변경 이유를 이름, 입력, 출력으로 드러내는 경계다.

```ts
const canCancelOrder = (order: Order) => {
  return order.status === "READY" && !order.invoiceIssued;
};
```

```tsx
const OrderCancelButton = ({
  disabled,
  onClick,
}: {
  disabled: boolean;
  onClick: () => void;
}) => {
  return (
    <Button disabled={disabled} onClick={onClick}>
      주문 취소
    </Button>
  );
};
```

`canCancelOrder`라는 이름은 업무 의미를 드러낸다. 이 함수는 "버튼 disabled 계산"이 아니라 "주문 취소 가능 여부"라는 도메인 규칙을 표현한다.

## 좋은 추상화의 조건

1. 이름이 의도를 드러낸다.
2. 입력과 출력이 명확하다.
3. 호출자가 몰라도 되는 세부사항을 숨긴다.
4. 호출자가 알아야 하는 도메인 의미는 드러낸다.
5. 변경 이유가 하나다.
6. 도메인 의미가 섞이지 않는다.
7. 같은 추상화 레벨끼리 조합된다.

## 추상화 레벨을 정렬한다

좋은 추상화는 단순히 작은 함수와 hook을 많이 만드는 것이 아니다.

낮은 레벨의 React state, 서버 query, UI side effect, 도메인 계산, 화면 orchestration이 하나의 hook 안에 섞이면 코드를 파일로 분리했더라도 좋은 추상화가 아니다.

좋은 추상화는 같은 레벨의 책임을 가진 단위들이 모이고, 상위 레벨은 하위 레벨을 조합해서 더 큰 유스케이스를 만든다.

## 추상화 레벨 예시

| 레벨 | 책임 | 예시 |
| --- | --- | --- |
| UI Primitive | 도메인 없는 UI 표현 | `Button`, `Dialog`, `BaseGrid` |
| Domain Display | 도메인 값 표현 | `OrderStatusBadge`, `InvoicePrintButton` |
| Pure Domain Helper | 순수 도메인 판단/계산 | `canCancelOrder`, `getPendingOrderCount` |
| Mapper/Serializer | 외부 DTO와 내부 모델 변환 | `mapOrderResponseToOrder`, `createOrderPayload` |
| API Function | 순수 HTTP 요청 | `fetchOrderList`, `cancelOrder` |
| Query/Mutation Hook | 서버 상태 생명주기 | `useOrderListQuery`, `useCancelOrderMutation` |
| Use-case Hook | 화면 유스케이스 조합 | `useOrderList`, `useOutboundWaitList` |
| Page/Feature Entry | 화면 조립 | `OrderPage`, `OutboundListPage` |

## 추상화 레벨과 SRP/DIP

추상화 레벨을 지키는 것은 프론트엔드에서 SRP와 DIP를 실천하는 방법이다.

### SRP

SRP는 "한 함수는 한 가지 일만 한다"가 아니라 "하나의 모듈은 하나의 변경 이유만 가진다"는 원칙이다.

query hook 안에 서버 조회, grid 제어, page loading, alert, derived state, legacy adapter가 모두 들어가면 변경 이유가 여러 개가 된다.

예를 들어 하나의 hook이 아래 책임을 모두 가진다면 SRP가 깨진다.

```ts
useQuery(...);
gridRef.current?.api?.deselectAll?.();
usePageLoading(...);
error(...);
setPendingCnt(...);
returnLegacyBodyShape(...);
```

이 hook은 다음 변경에 모두 영향을 받는다.

- API 응답 구조 변경
- grid 선택 초기화 정책 변경
- page loading 정책 변경
- alert 문구 변경
- pending count 계산 기준 변경
- legacy adapter 제거

따라서 서버 조회는 query hook, 도메인 계산은 pure helper, 화면 부수효과는 use-case hook, 렌더링은 component로 나눈다.

```ts
useOutboundWaitListQuery; // 서버 상태 조회
getPendingOrderCount; // 도메인 계산
useOutboundWaitList; // 화면 유스케이스 조립
OutboundListPage; // 렌더링 조립
```

### DIP

DIP는 상위 정책이 하위 구현 세부사항에 직접 의존하지 않게 하는 원칙이다.

프론트엔드에서 상위 UI는 API 응답 DTO, axios, queryClient, grid api, localStorage 같은 세부 구현에 직접 의존하지 않아야 한다.

컴포넌트는 hook이 제공하는 의미 있는 값과 이벤트에 의존한다.

### Do

```tsx
const OutboundListPage = () => {
  const {
    rows,
    pendingCount,
    refresh,
    clearSelection,
    isLoading,
  } = useOutboundWaitList();

  return (
    <OutboundTable
      rows={rows}
      pendingCount={pendingCount}
      loading={isLoading}
      onRefresh={refresh}
      onClearSelection={clearSelection}
    />
  );
};
```

컴포넌트는 무엇을 렌더링하고 어떤 이벤트를 호출할지만 안다. API 호출 방식, query key, grid api, response shape는 모른다.

### Do Not

```tsx
const OutboundTable = ({ queryClient, gridRef, response }) => {
  const rows = response.body.list;
  const pendingCount = rows.filter((row) => row.orderFlag === "P").length;

  const refresh = () => {
    queryClient.invalidateQueries(["outbound"]);
    gridRef.current.api.deselectAll();
  };

  return <BaseGrid rows={rows} />;
};
```

컴포넌트가 React Query, legacy response shape, grid api, pending count 정책을 모두 알게 된다. 상위 UI가 하위 구현 세부사항에 직접 의존하므로 변경에 약해진다.

## 1. 같은 코드가 아니라 같은 책임을 추상화한다

### Do

```tsx
const OrderStatusBadge = ({ status }: { status: OrderStatus }) => {
  return <Badge color={status.color}>{status.label}</Badge>;
};

const InvoiceStatusBadge = ({ status }: { status: InvoiceStatus }) => {
  return <Badge color={status.color}>{status.label}</Badge>;
};
```

두 컴포넌트가 비슷해 보여도 주문 상태와 송장 상태는 변경 이유가 다르다.

### Do Not

```tsx
const StatusBadge = ({
  type,
  code,
}: {
  type: "order" | "invoice";
  code: string;
}) => {
  if (type === "order") {
    return <Badge>{getOrderStatusLabel(code)}</Badge>;
  }

  return <Badge>{getInvoiceStatusLabel(code)}</Badge>;
};
```

이 추상화는 주문 정책과 송장 정책을 하나의 컴포넌트에 섞는다.

## 2. 구현만 숨긴 추출은 추상화가 아니다

### Do Not

```ts
const getValue = (data: any) => {
  if (data.status === "10" && !data.inv_no) {
    return false;
  }

  return true;
};
```

```tsx
<Button disabled={getValue(order)}>취소</Button>
```

문제:

- 이름이 의도를 설명하지 않는다.
- 입력 타입이 불명확하다.
- 반환값의 의미가 드러나지 않는다.
- 도메인 규칙인지 UI 계산인지 알 수 없다.
- 호출자가 내부 구현을 알아야 올바르게 사용할 수 있다.

### Do

```ts
const canCancelOrder = (order: Order) => {
  return order.status === "READY" && !order.invoiceIssued;
};
```

```tsx
<OrderCancelButton
  disabled={!canCancelOrder(order)}
  onClick={() => cancelOrder(order.id)}
/>
```

장점:

- 함수 이름이 도메인 규칙을 설명한다.
- 입력과 출력이 명확하다.
- 정책 변경 위치가 분명하다.
- UI는 결과를 표현하기만 한다.

## 3. `useSomethingQuery`는 query 책임에 집중한다

`useSomethingQuery`라는 이름을 붙였다면 최소한 서버 상태 조회 책임에 집중해야 한다.

그 안에서 grid ref를 조작하거나, page loading을 띄우거나, alert을 열거나, legacy response shape를 맞추기 시작하면 조회 추상화가 아니라 화면 구현 일부를 숨긴 추출일 가능성이 높다.

### Do Not

```ts
const useOutboundWaitListQuery = ({ queryData, gridRef }: Params) => {
  const [tableData, setTableData] = useState<OutboundListRow[]>([]);
  const [pendingCnt, setPendingCnt] = useState(0);
  const { error } = useAlertDialog();

  const queryResult = useQuery(
    outboundListKeys.waitList(queryData),
    () => fetchOutboundWaitList(queryData),
    {
      enabled: queryData.sellerId !== undefined,
      onSuccess: (result) => {
        gridRef.current?.api?.deselectAll?.();
        const list = result.data.list;
        setTableData(list);
        setPendingCnt(list.filter((row) => row.orderFlag === "P").length);
      },
      onError: (caughtError) => {
        error({ content: getErrorMessage(caughtError) });
      },
    },
  );

  usePageLoading(queryResult.isLoading || queryResult.isFetching);

  return {
    tableData,
    pendingCnt,
    data: queryResult.data
      ? { body: { list: queryResult.data.data.list, totCnt: queryResult.data.data.totCnt } }
      : undefined,
    isLoading: queryResult.isLoading,
  };
};
```

이 hook은 이름상으로는 "출고 대기 목록 query"처럼 보이지만 실제로는 여러 레벨의 책임을 동시에 가진다.

| 섞인 책임 | 문제 |
| --- | --- |
| `useQuery` | 서버 상태 조회 |
| `gridRef.current.api.deselectAll()` | UI grid side effect |
| `usePageLoading` | page-level loading |
| `useAlertDialog` | 사용자 피드백 |
| `tableData`, `pendingCnt` state | query 결과에서 파생 가능한 local state |
| legacy `body` shape 반환 | 레거시 호출부 어댑터 |

이것은 추상화라기보다 여러 구현 세부사항을 hook 내부에 숨긴 추출에 가깝다.

### Do

```ts
const useOutboundWaitListQuery = (queryData: OutboundListQuery) => {
  return useQuery({
    queryKey: outboundListKeys.waitList(queryData),
    queryFn: () => fetchOutboundWaitList(queryData),
    enabled: queryData.sellerId !== undefined,
    select: (response) => response.data,
  });
};
```

```ts
const getPendingOrderCount = (rows: OutboundListRow[]) => {
  return rows.filter((row) => row.orderFlag === "P").length;
};
```

```ts
const useOutboundWaitList = ({ queryData, gridRef }: Params) => {
  const query = useOutboundWaitListQuery(queryData);
  const { error } = useAlertDialog();

  usePageLoading(query.isLoading || query.isFetching);

  useEffect(() => {
    if (query.isSuccess) {
      gridRef.current?.api?.deselectAll?.();
    }
  }, [query.isSuccess, gridRef]);

  useEffect(() => {
    if (query.error) {
      error({ content: getOutboundWaitListErrorMessage(query.error) });
    }
  }, [query.error, error]);

  const rows = query.data?.list ?? [];

  return {
    rows,
    pendingCount: getPendingOrderCount(rows),
    totalCount: query.data?.totCnt ?? 0,
    isLoading: query.isLoading,
  };
};
```

이 구조에서는 각 레벨이 분리된다.

| 단위 | 책임 |
| --- | --- |
| `useOutboundWaitListQuery` | 서버 상태 조회 |
| `getPendingOrderCount` | 순수 도메인 계산 |
| `useOutboundWaitList` | 화면 유스케이스 조합 |
| page/component | 결과 렌더링 |

## 4. props/options가 계속 늘어나면 추상화 실패 신호다

### Do

```tsx
const OrderCancelDialog = ({
  open,
  order,
  onClose,
  onConfirm,
}: {
  open: boolean;
  order: Order;
  onClose: () => void;
  onConfirm: () => void;
}) => {
  return (
    <ConfirmDialog
      open={open}
      title="주문 취소"
      content={`${order.orderNo} 주문을 취소하시겠습니까?`}
      onClose={onClose}
      onConfirm={onConfirm}
    />
  );
};
```

### Do Not

```tsx
const BusinessConfirmDialog = ({
  type,
  open,
  title,
  order,
  invoice,
  packing,
  cancelReason,
  showReasonInput,
  showWarning,
  warningMessage,
  onClose,
  onConfirm,
}: Props) => {
  // type에 따라 주문 취소, 송장 재출력, 포장 완료를 모두 처리
};
```

옵션이 늘어나는 공통 컴포넌트는 대부분 서로 다른 유스케이스를 억지로 묶고 있다는 신호다.

## 5. Hook 추상화는 유스케이스 단위로 한다

### Do

```ts
const useCancelOrder = () => {
  const queryClient = useQueryClient();

  return useMutation({
    mutationFn: cancelOrder,
    onSuccess: () => {
      queryClient.invalidateQueries({ queryKey: ["orders"] });
    },
  });
};
```

### Do Not

```ts
const useBusinessMutation = ({
  mutationFn,
  successMessage,
  errorMessage,
  invalidateKeys,
  redirectTo,
  showLoading,
  confirmBeforeRun,
}: Options) => {
  // 모든 mutation 공통 처리
};
```

공통 mutation hook이 커질수록 각 유스케이스의 정책이 options로 새어 나온다.

## 7. 추상화 이름은 구현이 아니라 의도를 드러낸다

### Do

```ts
const createShippingAddressPayload = (form: ShippingAddressFormValues) => {
  return {
    receiverName: form.receiverName.trim(),
    zipCode: form.zipCode,
    address1: form.address1,
    address2: form.address2,
  };
};
```

### Do Not

```ts
const convertData = (data: any) => {
  return {
    name: data.name.trim(),
    zip: data.zip,
    addr1: data.addr1,
    addr2: data.addr2,
  };
};
```

`convertData`, `formatData`, `handleData`, `getValue` 같은 이름은 추상화가 아니라 의미를 숨긴다.

## 8. 인터페이스는 가능한 한 표준을 따른다

좋은 추상화는 사용자가 새 규칙을 외우게 만들지 않는다.

기반 UI 컴포넌트는 HTML과 React에서 이미 익숙한 표준 인터페이스를 따른다. Input은 `value/onChange`, Checkbox는 `checked/onChange`, Select는 `value/onChange/options`를 기본으로 한다.

도메인 이름은 사용하는 쪽에 남기고, 기반 컴포넌트 안으로 새지 않게 한다.

### Do

```tsx
<AmountInput
  label="목표금액"
  value={goalValue}
  onChange={setGoalValue}
/>

<Select
  label="기간"
  value={selectedPeriod}
  onChange={setSelectedPeriod}
  options={periodOptions}
/>

<Checkbox
  checked={isAgreed}
  onChange={setIsAgreed}
/>
```

`AmountInput`은 금액 입력 UI만 알면 된다. 그 값이 목표금액인지, 결제금액인지, 배송비인지는 사용하는 쪽의 관심사다.

### Do Not

```tsx
<AmountInput
  label="목표금액"
  goalValue={goalValue}
  setGoalValue={setGoalValue}
/>

<Select
  label="기간"
  selectedPeriod={selectedPeriod}
  setSelectedPeriod={setSelectedPeriod}
/>

<Checkbox
  isAgreed={isAgreed}
  setIsAgreed={setIsAgreed}
/>
```

기반 UI 컴포넌트가 `goalValue`, `selectedPeriod`, `isAgreed` 같은 도메인 이름을 props로 받기 시작하면 재사용 가능한 UI 추상화가 아니라 특정 화면 구현물이 된다.

### 도메인 컴포넌트는 도메인 props를 받을 수 있다

표준 인터페이스를 따른다는 말이 모든 컴포넌트에서 도메인 props를 금지한다는 뜻은 아니다.

도메인 유스케이스 컴포넌트는 도메인 의미를 props로 드러낼 수 있다.

```tsx
<OrderCancelDialog
  open={open}
  order={order}
  onConfirm={cancelOrder}
  onClose={close}
/>
```

`OrderCancelDialog`는 주문 취소라는 유스케이스를 표현하는 컴포넌트이므로 `order`를 받는 것이 자연스럽다.

문제는 `AmountInput`, `TextInput`, `Select`, `Checkbox` 같은 기반 UI 컴포넌트가 특정 화면의 도메인 이름을 props로 받는 경우다.

### 커스텀 훅도 익숙한 인터페이스를 따른다

단순 상태 하나를 감싸는 hook은 `useState` 형태를 우선 고려한다.

```ts
const [view, setView] = useView();
```

```ts
export const useView = () => {
  const [view, setView] = useState<"product" | "result">("product");

  return [view, setView] as const;
};
```

이렇게 하면 내부 구현이 `useState`에서 Zustand, Jotai, Redux 등으로 바뀌어도 외부 사용 방식은 유지할 수 있다.

다만 여러 값과 여러 action을 함께 반환하는 use-case hook은 object return을 사용한다.

```ts
const {
  rows,
  pendingCount,
  refresh,
  cancelOrder,
} = useOutboundWaitList();
```

tuple은 단순 상태에는 익숙하지만, 반환값이 많아지면 순서 의존성이 커진다.

## 나쁜 추상화의 신호

- `getData`, `handleData`, `convertData`, `getValue`처럼 의도가 없는 이름
- `type`, `mode`, `variant` 분기가 계속 늘어남
- `any` 입력
- boolean option이 많음
- 호출자가 내부 구현을 알아야 제대로 쓸 수 있음
- 코드 줄 수는 줄었는데 변경은 더 어려워짐
- query hook 안에 UI side effect가 들어감
- feature 업무 규칙이 `shared/utils`로 올라감
- 서로 다른 도메인 정책이 하나의 공통 컴포넌트에 섞임

## 추상화 판단 기준

```md
1. 반복되는 코드가 같은 책임인가?
   -> 아니면 분리한다.

2. 같은 이유로 변경될 코드인가?
   -> 아니면 분리한다.

3. 도메인 의미가 같은가?
   -> 아니면 분리한다.

4. 추상화 후 이름이 자연스럽게 붙는가?
   -> 아니면 아직 추상화하지 않는다.

5. options, type, mode 분기가 늘어나는가?
   -> 추상화를 쪼갠다.

6. shared로 올려도 도메인 지식이 새지 않는가?
   -> 아니면 feature 내부에 둔다.

7. 같은 추상화 레벨끼리 조합되어 있는가?
   -> 아니면 레벨을 분리한다.

8. 파일로 뺐을 뿐 호출자가 여전히 내부 구현을 알아야 하는가?
   -> 추상화가 아니라 추출일 가능성이 높다.
```

## 결론

추상화의 목적은 코드를 안 보이게 만드는 것이 아니다.

추상화의 목적은 호출자가 몰라도 되는 세부사항을 감추고, 알아야 하는 의도를 드러내는 것이다.

중복은 추상화의 충분조건이 아니다. 반복이 같은 책임과 같은 변경 이유를 가질 때만 추상화한다.

파일로 뺐다고 추상화가 아니다. 같은 추상화 레벨의 책임을 정렬하고, 상위 레벨에서 하위 레벨을 조합할 수 있을 때 좋은 추상화라고 부를 수 있다.
