## Background

프론트엔드 페이지 마이그레이션 이후 회귀 검증을 수동 QA에 크게 의존하고 있다. 마이그레이션 대상 페이지가 늘어날수록 동일한 핵심 업무 흐름을 반복 검증해야 하고, 이 과정이 QA 병목으로 이어져 마이그레이션이 완료된 뒤에도 릴리즈가 지연되고 있다.

특히 CGKR UI는 출고, 입고, 재고 등 운영 업무 흐름이 화면 단위로 연결되어 있고, 레거시 구조에서 현대적인 feature 구조로 점진 이관 중이다. 이 과정에서 UI 렌더링, 검색 조건, 그리드 동작, API 연동, 권한/라우팅 같은 사용자 관점의 회귀가 발생할 수 있으므로, 단위 테스트만으로는 릴리즈 판단에 필요한 신뢰를 충분히 제공하기 어렵다.

따라서 페이지 마이그레이션마다 반복되는 핵심 사용자 흐름을 E2E 테스트로 고정해 수동 QA 범위를 줄이고, 릴리즈 전 회귀 검증을 자동화할 필요가 있다.

## Decision

프론트엔드 페이지 마이그레이션의 핵심 회귀 검증 수단으로 Playwright 기반 E2E 테스트를 도입한다.

초기 적용 범위는 모든 페이지의 완전 자동화가 아니라, 마이그레이션된 페이지의 주요 happy path와 릴리즈 차단 가능성이 높은 업무 흐름으로 제한한다. E2E 테스트는 수동 QA를 완전히 대체하기보다, 반복 회귀 검증을 자동화해 QA가 예외 케이스와 탐색적 검증에 집중할 수 있도록 한다.

CI에서는 PR 또는 배포 전 검증 단계에서 실행 가능한 형태로 구성하고, flaky test가 릴리즈 병목이 되지 않도록 테스트 데이터, selector 전략, 네트워크 mocking 또는 테스트 환경 의존성을 별도 기준으로 관리한다.

## Options

| Option | Pros | Cons |
| --- | --- | --- |
| 수동 QA 유지 | 추가 구현 비용이 없고 기존 프로세스를 유지할 수 있다. | 마이그레이션 페이지가 늘수록 QA 병목이 커지고, 반복 회귀 검증의 누락 가능성이 높다. |
| 단위/통합 테스트 중심 보강 | 실행 속도가 빠르고 특정 로직의 실패 원인을 좁히기 쉽다. | 실제 브라우저에서의 페이지 이동, 그리드 상호작용, 사용자 업무 흐름 회귀를 충분히 검증하기 어렵다. |
| Playwright 기반 E2E 테스트 도입 | 사용자 관점의 핵심 흐름을 자동 검증할 수 있고, 마이그레이션 이후 릴리즈 판단에 필요한 회귀 신뢰도를 높일 수 있다. | 테스트 환경과 데이터 관리가 필요하며, 부실하게 작성하면 flaky test와 CI 시간 증가가 발생할 수 있다. |

**최종 결정:** Playwright 기반 E2E 테스트를 도입한다. 반복 수동 QA 병목을 줄이는 것이 현재 마이그레이션 릴리즈의 핵심 제약이며, 사용자 관점의 회귀 검증은 E2E 테스트가 가장 직접적으로 해결할 수 있다.
