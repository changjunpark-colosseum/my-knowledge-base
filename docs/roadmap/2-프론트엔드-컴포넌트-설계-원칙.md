# 2. 프론트엔드 컴포넌트 설계 원칙

## 목적

컴포넌트의 책임을 명확히 분리해서 UI 변경이 비즈니스 로직을 깨지 않도록 한다.

컴포넌트는 사용자에게 도메인 상태를 표현하고 입력 이벤트를 전달하는 역할을 한다. 비즈니스 규칙, API 호출, 서버 상태 관리, 데이터 변환은 hook, api, mapper, domain layer로 분리한다.

## 핵심 문장

컴포넌트는 도메인을 표현할 수 있지만, 도메인 정책을 소유해서는 안 된다.

`OrderStatusBadge`, `InvoicePrintButton`, `ShippingAddressForm`처럼 도메인 이름을 가진 컴포넌트는 괜찮다. 하지만 그 안에서 API 호출, 서버 DTO 해석, 권한 판단, 업무 정책 조건을 직접 처리하면 안 된다.

도메인 판단은 hook, mapper, domain helper에 두고 컴포넌트는 결과를 표현한다.

## 핵심 원칙

1. 컴포넌트는 UI 표현에 집중한다.
2. 비즈니스 로직은 hook으로 분리한다.
3. API 호출은 컴포넌트에서 직접 하지 않는다.
4. 서버 응답 DTO를 컴포넌트에서 직접 가공하지 않는다.
5. 도메인 컴포넌트라도 정책 판단을 직접 소유하지 않는다.
6. 재사용성보다 책임 분리를 먼저 고려한다.
7. 컴포넌트 props는 UI에 필요한 값과 이벤트로 제한한다.
8. 컴포넌트는 자신이 렌더링하는 데 필요한 최소한의 사실만 알아야 한다.

## 컴포넌트 분류

| 분류 | 책임 | 예시 |
| --- | --- | --- |
| Page Component | 화면 조립, feature 연결 | `OrderPage` |
| Feature Component | 특정 유스케이스 UI 표현 | `OrderCancelDialog` |
| Domain Display Component | 도메인 값 표현 | `OrderStatusBadge` |
| Form Component | 입력 UI와 submit 이벤트 | `OrderSearchForm` |
| Base UI Component | 도메인 없는 공통 UI | `BaseGrid`, `Dialog`, `Button` |

## 책임 위치

| 책임 | 위치 |
| --- | --- |
| API 요청 함수 | `api/` |
| 서버 DTO 스키마 | `schema/` |
| 서버 DTO -> 내부 모델 변환 | `mapper`, `lib`, `domain helper` |
| 서버 상태 조회/변경 | `hooks/` 안의 React Query |
| 업무 규칙 판단 | `hooks/`, `model`, `domain helper` |
| 화면 표현 | `components/` |
| 화면 조립 | page, feature entry component |

## 1. API 호출은 컴포넌트에서 하지 않는다

### Do

```tsx
const OrderList = () => {
  const { orders, isLoading, cancelOrder } = useOrderList();

  if (isLoading) {
    return <Loading />;
  }

  return <OrderTable orders={orders} onCancelOrder={cancelOrder} />;
};
```

```ts
const useOrderList = () => {
  const queryClient = useQueryClient();

  const { data, isLoading } = useQuery({
    queryKey: ["orders"],
    queryFn: fetchOrderList,
  });

  const cancelOrderMutation = useMutation({
    mutationFn: cancelOrder,
    onSuccess: () => {
      queryClient.invalidateQueries({ queryKey: ["orders"] });
    },
  });

  return {
    orders: data?.data.orders ?? [],
    isLoading,
    cancelOrder: cancelOrderMutation.mutate,
  };
};
```

### Do Not

```tsx
const OrderTable = () => {
  const [orders, setOrders] = useState<Order[]>([]);

  useEffect(() => {
    fetchOrderList().then((response) => {
      setOrders(response.data.orders);
    });
  }, []);

  const handleCancel = async (orderId: number) => {
    await cancelOrder(orderId);
    const response = await fetchOrderList();
    setOrders(response.data.orders);
  };

  return <BaseGrid rows={orders} />;
};
```

컴포넌트가 API 호출, loading state, refetch, mutation 후 갱신까지 직접 처리하면 UI와 서버 상태 관리 책임이 결합된다.

## 2. 도메인 컴포넌트라도 정책 판단을 직접 하지 않는다

도메인 이름을 가진 컴포넌트는 허용한다. 하지만 업무 정책을 직접 소유하면 안 된다.

### Do

```tsx
const OrderCancelButton = ({
  disabled,
  onClick,
}: {
  disabled: boolean;
  onClick: () => void;
}) => {
  return (
    <Button disabled={disabled} onClick={onClick}>
      주문 취소
    </Button>
  );
};
```

```ts
const useOrderActions = (order: Order) => {
  const canCancel = order.status === "READY" && !order.invoiceIssued;

  return {
    canCancel,
    cancelOrder,
  };
};
```

```tsx
const OrderRowActions = ({ order }: { order: Order }) => {
  const { canCancel, cancelOrder } = useOrderActions(order);

  return (
    <OrderCancelButton
      disabled={!canCancel}
      onClick={() => cancelOrder(order.id)}
    />
  );
};
```

### Do Not

```tsx
const OrderCancelButton = ({ order }: { order: Order }) => {
  const disabled = order.status !== "READY" || order.invoiceIssued;

  return <Button disabled={disabled}>주문 취소</Button>;
};
```

컴포넌트가 `주문 취소 가능 조건`을 직접 알기 시작하면 정책 변경 시 UI 컴포넌트를 수정해야 한다.

## 3. 서버 DTO를 컴포넌트에서 직접 해석하지 않는다

컴포넌트는 서버 응답 필드명이나 코드값을 직접 알지 않는 것이 좋다.

### Do

```tsx
const OrderStatusBadge = ({ status }: { status: OrderStatus }) => {
  return <Badge color={status.color}>{status.label}</Badge>;
};
```

```ts
const mapOrderResponseToOrder = (response: OrderResponse): Order => {
  return {
    id: response.ord_no,
    status: {
      code: response.ord_status,
      label: getOrderStatusLabel(response.ord_status),
      color: getOrderStatusColor(response.ord_status),
    },
  };
};
```

### Do Not

```tsx
const OrderStatusBadge = ({ order }: { order: OrderResponse }) => {
  if (order.ord_status === "10") {
    return <Badge color="default">주문 접수</Badge>;
  }

  if (order.ord_status === "20") {
    return <Badge color="primary">출고 준비</Badge>;
  }

  return <Badge>알 수 없음</Badge>;
};
```

컴포넌트가 서버 코드값을 직접 알면 API 변경과 UI 변경이 강하게 결합된다.

## 4. 컴포넌트 props는 표현 값과 이벤트로 제한한다

컴포넌트는 화면에 필요한 값과 사용자 이벤트만 받는 것이 좋다.

### Do

```tsx
const OrderSummary = ({
  totalCount,
  completedCount,
  onRefresh,
}: {
  totalCount: number;
  completedCount: number;
  onRefresh: () => void;
}) => {
  return (
    <SummaryCard
      totalCount={totalCount}
      completedCount={completedCount}
      onRefresh={onRefresh}
    />
  );
};
```

### Do Not

```tsx
const OrderSummary = ({
  queryClient,
  searchParams,
  orderApi,
}: {
  queryClient: QueryClient;
  searchParams: OrderSearchParams;
  orderApi: OrderApi;
}) => {
  const handleRefresh = async () => {
    await queryClient.invalidateQueries(["orders", searchParams]);
  };

  return <SummaryCard onRefresh={handleRefresh} />;
};
```

컴포넌트에 infra 의존성이 들어오면 UI 테스트와 재사용이 어려워진다.

## 5. 컴포넌트에서 복잡한 데이터 변환을 하지 않는다

렌더링에 필요한 단순 format은 허용할 수 있지만, 서버 DTO 변환이나 업무 의미가 있는 가공은 컴포넌트 밖으로 뺀다.

### Do

```tsx
const OrderTable = ({ rows }: { rows: OrderTableRow[] }) => {
  return <BaseGrid rows={rows} columns={orderColumns} />;
};
```

```ts
const createOrderTableRows = (orders: Order[]): OrderTableRow[] => {
  return orders.map((order) => ({
    id: order.id,
    customerName: order.customerName,
    statusLabel: order.status.label,
    invoiceNo: order.invoiceNo ?? "-",
  }));
};
```

### Do Not

```tsx
const OrderTable = ({ orders }: { orders: OrderResponse[] }) => {
  const rows = orders.map((order) => ({
    id: order.ord_no,
    customerName: order.cust_nm,
    statusLabel: order.ord_status === "10" ? "주문 접수" : "출고 준비",
    invoiceNo: order.inv_no || "-",
  }));

  return <BaseGrid rows={rows} columns={orderColumns} />;
};
```

서버 필드명, 코드값, fallback 정책이 컴포넌트에 섞이면 UI 변경과 데이터 계약 변경을 분리하기 어렵다.

## 6. 필터링과 정렬은 표현 컴포넌트 밖에서 결정한다

필터링, 정렬, 노출 조건 같은 도메인 로직은 표현 컴포넌트 안에 직접 두지 않는다.

표현 컴포넌트는 "무엇을 보여줄지"가 이미 결정된 데이터를 받아 렌더링한다. "어떤 기준으로 걸러낼지", "어떤 순서로 보여줄지", "어떤 상품이나 주문을 노출할지"는 use-case layer에서 결정한다.

### Do

```ts
const filterByMinimumAmount = (product: Product) => {
  return product.price >= 10000;
};

const filterInStock = (product: Product) => {
  return product.stockQty > 0;
};

const orderByHighPrice = (a: Product, b: Product) => {
  return b.price - a.price;
};
```

```ts
const useProductList = ({
  filters,
  orderBy,
}: {
  filters: Array<(product: Product) => boolean>;
  orderBy?: (a: Product, b: Product) => number;
}) => {
  const { data, isLoading } = useQuery({
    queryKey: ["products"],
    queryFn: fetchProducts,
    select: (products) => {
      const filteredProducts = filters.reduce(
        (items, filter) => items.filter(filter),
        products,
      );

      if (!orderBy) {
        return filteredProducts;
      }

      return [...filteredProducts].sort(orderBy);
    },
  });

  return {
    products: data ?? [],
    isLoading,
  };
};
```

```tsx
const ProductListContainer = () => {
  const { products, isLoading } = useProductList({
    filters: [filterByMinimumAmount, filterInStock],
    orderBy: orderByHighPrice,
  });

  return <ProductList products={products} loading={isLoading} />;
};
```

```tsx
const ProductList = ({
  products,
  loading,
}: {
  products: Product[];
  loading: boolean;
}) => {
  if (loading) {
    return <Loading />;
  }

  return (
    <ul>
      {products.map((product) => (
        <li key={product.id}>
          {product.name} - {product.price}
        </li>
      ))}
    </ul>
  );
};
```

### Do Not

```tsx
const ProductList = () => {
  const { data = [], isLoading } = useQuery({
    queryKey: ["products"],
    queryFn: fetchProducts,
  });

  const products = data
    .filter((product) => product.price >= 10000)
    .filter((product) => product.stockQty > 0)
    .sort((a, b) => b.price - a.price);

  if (isLoading) {
    return <Loading />;
  }

  return (
    <ul>
      {products.map((product) => (
        <li key={product.id}>
          {product.name} - {product.price}
        </li>
      ))}
    </ul>
  );
};
```

이 방식은 `ProductList`가 데이터 조회, 필터 정책, 정렬 정책, 화면 표현을 모두 소유하게 만든다. 상품 가격순, 재고 있는 상품만, 카테고리별 노출처럼 UI 목적이 달라질 때마다 표현 컴포넌트를 수정해야 한다.

### 기준

```md
도메인 로직은 부모 컴포넌트에 무조건 올리는 것이 아니다.
도메인 로직은 use-case layer에서 결정하고, 표현 컴포넌트는 결정된 결과만 렌더링한다.
```

| 책임 | 위치 |
| --- | --- |
| 필터 조건 함수 | domain helper, lib |
| 정렬 기준 함수 | domain helper, lib |
| 필터/정렬 조합 | hook, container, page |
| 서버 데이터 조회와 select 가공 | hook |
| 목록 렌더링 | 표현 컴포넌트 |

## 7. Form 컴포넌트는 입력과 submit 이벤트에 집중한다

Form 컴포넌트는 입력값을 수집하고 submit 이벤트를 전달한다. 검색 실행, API 호출, URL 변경은 상위 hook 또는 page에서 처리한다.

### Do

```tsx
const OrderSearchForm = ({
  defaultValues,
  onSubmit,
}: {
  defaultValues: OrderSearchFormValues;
  onSubmit: (values: OrderSearchFormValues) => void;
}) => {
  const form = useForm<OrderSearchFormValues>({
    defaultValues,
  });

  return (
    <form onSubmit={form.handleSubmit(onSubmit)}>
      <TextField {...form.register("keyword")} />
      <Button type="submit">검색</Button>
    </form>
  );
};
```

```tsx
const OrderPage = () => {
  const { params, setSearchParams } = useOrderSearchParams();
  const { orders } = useOrderList(params);

  return (
    <>
      <OrderSearchForm
        defaultValues={params}
        onSubmit={(values) => setSearchParams({ ...values, page: 1 })}
      />
      <OrderTable orders={orders} />
    </>
  );
};
```

### Do Not

```tsx
const OrderSearchForm = () => {
  const form = useForm<OrderSearchFormValues>();
  const queryClient = useQueryClient();
  const router = useRouter();

  const onSubmit = async (values: OrderSearchFormValues) => {
    router.push({ query: values });
    await queryClient.invalidateQueries(["orders"]);
  };

  return (
    <form onSubmit={form.handleSubmit(onSubmit)}>
      <TextField {...form.register("keyword")} />
      <Button type="submit">검색</Button>
    </form>
  );
};
```

Form 컴포넌트가 URL 변경, Query 무효화, 검색 실행을 모두 알면 재사용이 어려워진다.

## 8. Page는 조립하고, Feature는 유스케이스를 표현한다

Page Component는 여러 feature와 hook을 연결하는 조립 지점이다. 단, Page 안에 비즈니스 로직을 길게 직접 작성하지 않는다.

### Do

```tsx
const OrderPage = () => {
  const { params, setSearchParams } = useOrderSearchParams();
  const { orders, isLoading, cancelOrder } = useOrderList(params);

  return (
    <>
      <OrderSearchForm
        defaultValues={params}
        onSubmit={(values) => setSearchParams({ ...values, page: 1 })}
      />
      <OrderTable
        orders={orders}
        loading={isLoading}
        onCancelOrder={cancelOrder}
      />
    </>
  );
};
```

### Do Not

```tsx
const OrderPage = () => {
  const [orders, setOrders] = useState<Order[]>([]);
  const [loading, setLoading] = useState(false);
  const [keyword, setKeyword] = useState("");

  const search = async () => {
    setLoading(true);
    const response = await fetchOrderList({ keyword });
    const mappedOrders = response.data.orders.map((order) => ({
      ...order,
      statusLabel: order.ord_status === "10" ? "주문 접수" : "출고 준비",
    }));
    setOrders(mappedOrders);
    setLoading(false);
  };

  return <>{/* UI 생략 */}</>;
};
```

Page가 조립 지점을 넘어 데이터 조회, 변환, 정책 판단, loading 처리를 모두 직접 가지면 변경 이유가 많아진다.

## 9. 재사용성보다 책임 분리를 먼저 본다

처음부터 모든 컴포넌트를 공통화하려고 하지 않는다. 책임이 다르면 비슷해 보여도 분리한다.

### Do

```tsx
const OrderStatusBadge = ({ status }: { status: OrderStatus }) => {
  return <Badge color={status.color}>{status.label}</Badge>;
};

const InvoiceStatusBadge = ({ status }: { status: InvoiceStatus }) => {
  return <Badge color={status.color}>{status.label}</Badge>;
};
```

### Do Not

```tsx
const StatusBadge = ({
  type,
  code,
}: {
  type: "order" | "invoice";
  code: string;
}) => {
  const label =
    type === "order" ? getOrderStatusLabel(code) : getInvoiceStatusLabel(code);

  return <Badge>{label}</Badge>;
};
```

겉모양이 비슷하다는 이유로 합치면 서로 다른 도메인 정책이 하나의 공통 컴포넌트에 섞인다.

## 10. 컴포넌트가 몰라야 하는 것

컴포넌트에 아래 코드가 보이면 책임이 섞였는지 의심한다.

- `useQuery`
- `useMutation`
- `apiClient`
- `axios`
- `fetchSomething`
- `queryClient`
- `localStorage`
- 서버 DTO 필드명
- 권한/상태 정책 조건문
- 복잡한 데이터 변환 로직

단, Page 또는 feature entry component에서 hook을 호출해 화면을 조립하는 것은 허용한다.

## 11. 판단 기준

컴포넌트 설계가 애매하면 다음 순서로 판단한다.

```md
1. 이 로직이 화면 표현만을 위한 것인가?
   -> 컴포넌트에 둘 수 있다.

2. 서버 데이터 조회/변경과 관련 있는가?
   -> hook과 api로 분리한다.

3. 서버 DTO 필드명이나 코드값을 해석하는가?
   -> schema, mapper, domain helper로 분리한다.

4. 업무 정책이나 권한 판단인가?
   -> hook, model, domain helper로 분리한다.

5. 입력값을 수집하는 폼인가?
   -> Form 컴포넌트는 입력과 submit 이벤트만 책임진다.

6. 여러 도메인에서 비슷하게 생겼는가?
   -> 바로 공통화하지 말고 책임이 같은지 먼저 확인한다.
```

## 금지 패턴 요약

### 컴포넌트에서 API 직접 호출

```tsx
useEffect(() => {
  fetchOrderList().then(setOrders);
}, []);
```

서버 상태는 React Query 기반 hook으로 관리한다.

### 컴포넌트에서 업무 정책 판단

```tsx
const disabled = order.status !== "READY" || order.invoiceIssued;
```

정책 판단은 hook 또는 domain helper로 분리한다.

### 컴포넌트에서 서버 DTO 직접 해석

```tsx
order.ord_status === "10" ? "주문 접수" : "출고 준비";
```

서버 DTO는 mapper를 통해 내부 모델로 변환한 뒤 컴포넌트에 전달한다.

### 컴포넌트 props로 infra 의존성 전달

```tsx
<OrderSummary queryClient={queryClient} orderApi={orderApi} />
```

컴포넌트 props는 표현 값과 이벤트로 제한한다.

## 결론

도메인 표현 컴포넌트는 허용한다. 도메인 정책 소유 컴포넌트는 지양한다.

좋은 컴포넌트는 많은 일을 하는 컴포넌트가 아니라, 자신이 무엇을 몰라도 되는지 명확한 컴포넌트다.
