# 1. 프론트엔드 상태관리 전략

## 목적

프론트엔드 상태관리는 특정 라이브러리를 고르는 문제가 아니라, 상태의 원본 소유자와 생명주기를 구분하는 문제다.

상태를 한 곳에 몰아넣지 않고 성격에 따라 분리해서 관리한다. 이를 통해 불필요한 전역 상태, 중복 API 캐시, URL과 UI 상태 불일치, 새로고침 시 상태 유실 문제를 줄인다.

## 상태 분류

| 분류 | 원본 소유자 | 대표 도구 | 예시 |
| --- | --- | --- | --- |
| Local State | 현재 컴포넌트 또는 화면 | `useState`, `useReducer` | 모달 열림, 선택 row, 탭 |
| Form State | 입력 폼 | React Hook Form | 검색 조건 입력값, 등록/수정 폼 |
| Global State | 앱 클라이언트 맥락 | MobX, 제한적 Context | 로그인 사용자, 권한, 선택 창고, 언어 |
| Server State | 서버 | React Query | 주문 목록, 재고, 출고 상세, 송장 정보 |
| URL State | 브라우저 URL | Next.js router query | 검색 조건, 페이지, 정렬, 필터, 상세 ID |
| Optimistic State | 임시 UI 반영 후 서버 확정 | React Query mutation lifecycle | 즐겨찾기 토글, 즉시 반영 가능한 상태 변경 |
| Derived State | 기존 상태에서 계산 | 변수, `useMemo` | 합계, 필터링 결과, 선택 개수 |

## 상태 선택 기준

| 질문 | 선택 |
| --- | --- |
| 컴포넌트 안에서만 쓰는가? | Local State |
| 입력 중인 폼 값인가? | Form State |
| 서버가 원본 데이터인가? | Server State |
| 새로고침, 링크 공유, 뒤로가기에 유지되어야 하는가? | URL State |
| 여러 페이지에서 공유하는 앱 설정이나 사용자 맥락인가? | Global State |
| 서버 응답 전 UI에 먼저 반영해야 하는가? | Optimistic State |
| 다른 상태에서 계산 가능한가? | State로 만들지 않는다 |

## 기본 원칙

1. 상태는 가능한 한 사용하는 곳 가까이에 둔다.
2. 서버 데이터는 React Query로 관리한다.
3. 전역 상태는 최소화한다.
4. 검색 조건과 페이지 상태는 URL에 둔다.
5. 같은 상태를 두 곳 이상에 복사하지 않는다.
6. Props Drilling은 Global State 도입 신호가 아니라 컴포넌트 구조 재검토 신호다.
7. 상태 변경의 원본 소유자를 명확히 한다.

## 1. Local State

컴포넌트 내부 또는 한 화면 안에서만 의미 있는 상태다.

### Do

```tsx
const OrderSearchPanel = () => {
  const [isDetailOpen, setIsDetailOpen] = useState(false);
  const [selectedOrderId, setSelectedOrderId] = useState<number | null>(null);

  return (
    <>
      <OrderTable onSelectOrder={setSelectedOrderId} />

      <OrderDetailDialog
        open={isDetailOpen}
        orderId={selectedOrderId}
        onClose={() => setIsDetailOpen(false)}
      />
    </>
  );
};
```

### Do Not

```tsx
// 특정 화면에서만 쓰는 모달 상태를 전역 store로 관리하지 않는다.
modalStore.open("orderDetail");
modalStore.setSelectedOrderId(orderId);
```

## 2. Form State

입력 중인 값의 묶음은 Form State로 관리한다.

### Do

```tsx
const OrderSearchForm = ({
  defaultValues,
  onSubmit,
}: {
  defaultValues: OrderSearchFormValues;
  onSubmit: (values: OrderSearchFormValues) => void;
}) => {
  const form = useForm<OrderSearchFormValues>({
    defaultValues,
  });

  return (
    <form onSubmit={form.handleSubmit(onSubmit)}>
      <TextField {...form.register("keyword")} />
      <Button type="submit">검색</Button>
    </form>
  );
};
```

### Do Not

```tsx
const OrderSearchForm = () => {
  const [keyword, setKeyword] = useState("");
  const [status, setStatus] = useState("");
  const [dateFrom, setDateFrom] = useState("");
  const [dateTo, setDateTo] = useState("");

  return <>{/* 입력 필드가 늘어날수록 초기화, 검증, submit 관리가 어려워진다. */}</>;
};
```

## 3. Global State

여러 화면에서 공유해야 하는 클라이언트 상태만 Global State로 둔다.

### Do

```ts
class UserStore {
  userId = "";
  userName = "";
  warehouseId = "";
  language = "ko";

  setWarehouseId(warehouseId: string) {
    this.warehouseId = warehouseId;
  }
}
```

```tsx
const HeaderWarehouseSelector = observer(() => {
  return (
    <Select
      value={userStore.warehouseId}
      onChange={(event) => userStore.setWarehouseId(String(event.target.value))}
    />
  );
});
```

### Do Not

```ts
class OrderStore {
  // 서버에서 조회한 목록 데이터를 전역 store에 복사하지 않는다.
  orders: Order[] = [];

  setOrders(orders: Order[]) {
    this.orders = orders;
  }
}
```

```tsx
const fetchOrders = async () => {
  const response = await fetchOrderList(params);
  orderStore.setOrders(response.data.orders);
};
```

## 4. Server State

서버가 원본인 데이터는 React Query로 관리한다.

### Do

```tsx
const useOrderList = (params: OrderListParams) => {
  return useQuery({
    queryKey: ["orders", params],
    queryFn: () => fetchOrderList(params),
    keepPreviousData: true,
  });
};
```

```tsx
const OrderList = ({ params }: { params: OrderListParams }) => {
  const { data, isLoading } = useOrderList(params);

  if (isLoading) {
    return <Loading />;
  }

  return <BaseGrid rows={data?.data.orders ?? []} />;
};
```

### Do Not

```tsx
const OrderList = ({ params }: { params: OrderListParams }) => {
  const [orders, setOrders] = useState<Order[]>([]);

  useEffect(() => {
    fetchOrderList(params).then((response) => {
      setOrders(response.data.orders);
    });
  }, [params]);

  return <BaseGrid rows={orders} />;
};
```

## 5. Mutation 후 갱신

서버 데이터를 변경한 뒤에는 관련 Query를 invalidate한다.

### Do

```tsx
const useCancelOrder = () => {
  const queryClient = useQueryClient();

  return useMutation({
    mutationFn: cancelOrder,
    onSuccess: () => {
      queryClient.invalidateQueries({ queryKey: ["orders"] });
    },
  });
};
```

### Do Not

```tsx
const cancel = async (orderId: number) => {
  await cancelOrder(orderId);

  // 화면 state를 수동으로 맞추기 시작하면 서버 상태와 어긋나기 쉽다.
  setOrders((prev) => prev.filter((order) => order.id !== orderId));
};
```

## 6. URL State

검색 조건, 페이지 번호, 정렬, 필터처럼 공유/복원되어야 하는 상태는 URL에 둔다.

### Do

```tsx
const useOrderSearchParams = () => {
  const router = useRouter();

  const page = Number(router.query.page ?? 1);
  const keyword = String(router.query.keyword ?? "");
  const status = String(router.query.status ?? "");

  const setSearchParams = (nextParams: {
    page?: number;
    keyword?: string;
    status?: string;
  }) => {
    router.push({
      pathname: router.pathname,
      query: {
        ...router.query,
        ...nextParams,
      },
    });
  };

  return {
    params: { page, keyword, status },
    setSearchParams,
  };
};
```

```tsx
const OrderPage = () => {
  const { params, setSearchParams } = useOrderSearchParams();
  const { data } = useOrderList(params);

  return (
    <OrderSearchForm
      defaultValues={params}
      onSubmit={(values) => {
        setSearchParams({
          ...values,
          page: 1,
        });
      }}
    />
  );
};
```

### Do Not

```tsx
const OrderPage = () => {
  const router = useRouter();

  const [keyword, setKeyword] = useState(String(router.query.keyword ?? ""));
  const [page, setPage] = useState(Number(router.query.page ?? 1));

  return <OrderSearchForm keyword={keyword} page={page} />;
};
```

URL에도 있고 Local State에도 있으면 원본이 둘이 된다.

## 7. Optimistic State

사용자 피드백이 즉시 필요하고 실패 시 되돌릴 수 있는 경우에만 사용한다.

### Do

```tsx
const useToggleFavoriteOrder = () => {
  const queryClient = useQueryClient();

  return useMutation({
    mutationFn: toggleFavoriteOrder,

    onMutate: async ({ orderId }) => {
      await queryClient.cancelQueries({ queryKey: ["orders"] });

      const previousOrders = queryClient.getQueryData<OrderListResponse>(["orders"]);

      queryClient.setQueryData<OrderListResponse>(["orders"], (current) => {
        if (!current) {
          return current;
        }

        return {
          ...current,
          orders: current.orders.map((order) =>
            order.id === orderId ? { ...order, favorite: !order.favorite } : order,
          ),
        };
      });

      return { previousOrders };
    },

    onError: (_error, _variables, context) => {
      if (context?.previousOrders) {
        queryClient.setQueryData(["orders"], context.previousOrders);
      }
    },

    onSettled: () => {
      queryClient.invalidateQueries({ queryKey: ["orders"] });
    },
  });
};
```

### Do Not

```tsx
const toggleFavorite = async (orderId: number) => {
  setOrders((prev) =>
    prev.map((order) =>
      order.id === orderId ? { ...order, favorite: !order.favorite } : order,
    ),
  );

  await toggleFavoriteOrder({ orderId });
};
```

실패 시 rollback이 없다면 Optimistic Update를 사용하지 않는다.

## 8. Derived State

기존 상태에서 계산 가능한 값은 별도 state로 만들지 않는다.

### Do

```tsx
const OrderSummary = ({ orders }: { orders: Order[] }) => {
  const completedCount = orders.filter((order) => order.status === "COMPLETED").length;
  const pendingCount = orders.filter((order) => order.status === "PENDING").length;

  return <Summary completedCount={completedCount} pendingCount={pendingCount} />;
};
```

### Do Not

```tsx
const OrderSummary = ({ orders }: { orders: Order[] }) => {
  const [completedCount, setCompletedCount] = useState(0);

  useEffect(() => {
    setCompletedCount(orders.filter((order) => order.status === "COMPLETED").length);
  }, [orders]);

  return <Summary completedCount={completedCount} />;
};
```

## 9. Props Drilling vs Composition / Context

Props Drilling이 발생한다고 해서 곧바로 Global State로 올리지 않는다.

Props Drilling은 보통 상태관리 도구가 부족해서 생기는 문제가 아니라, 컴포넌트 계층과 의존성 전달 방식이 현재 유스케이스에 맞지 않는다는 신호다.

### 원칙

1. 같은 화면 안에서만 쓰는 상태는 Global State로 올리지 않는다.
2. 중간 컴포넌트가 사용하지 않는 props를 계속 전달한다면 구조를 재검토한다.
3. 우선 컴포넌트 합성으로 의존성을 사용하는 위치에 가깝게 옮긴다.
4. 여러 하위 컴포넌트가 같은 화면 범위 안에서 상태를 공유해야 한다면 화면 전용 Context를 사용한다.
5. Context는 앱 전역 공유 수단이 아니라 특정 UI 레이어의 의존성 주입 경계로 사용한다.

### Do Not: Props Drilling을 Global State로 해결

```ts
class OrderSelectionStore {
  selectedOrderId: number | null = null;

  setSelectedOrderId(orderId: number) {
    this.selectedOrderId = orderId;
  }
}
```

```tsx
const OrderPage = observer(() => {
  return (
    <>
      <OrderTable
        selectedOrderId={orderSelectionStore.selectedOrderId}
        onSelectOrder={orderSelectionStore.setSelectedOrderId}
      />

      <OrderDetail orderId={orderSelectionStore.selectedOrderId} />
    </>
  );
});
```

이 방식은 선택 상태의 생명주기를 앱 전역으로 넓힌다. 결과적으로 다른 화면 진입 시 이전 선택값이 남거나, 테스트/초기화/추적이 어려워질 수 있다.

### Do Not: 중간 컴포넌트가 쓰지 않는 props 전달

```tsx
const OrderPage = () => {
  const [selectedOrderId, setSelectedOrderId] = useState<number | null>(null);

  return (
    <OrderLayout
      selectedOrderId={selectedOrderId}
      onSelectOrder={setSelectedOrderId}
    />
  );
};

const OrderLayout = ({
  selectedOrderId,
  onSelectOrder,
}: {
  selectedOrderId: number | null;
  onSelectOrder: (orderId: number) => void;
}) => {
  return (
    <OrderContent
      selectedOrderId={selectedOrderId}
      onSelectOrder={onSelectOrder}
    />
  );
};

const OrderContent = ({
  selectedOrderId,
  onSelectOrder,
}: {
  selectedOrderId: number | null;
  onSelectOrder: (orderId: number) => void;
}) => {
  return (
    <>
      <OrderTable onSelectOrder={onSelectOrder} />
      <OrderDetail orderId={selectedOrderId} />
    </>
  );
};
```

`OrderLayout`은 `selectedOrderId`, `onSelectOrder`를 사용하지 않는다. 이런 props 전달이 반복되면 컴포넌트 계층이 상태 전달 경로에 끌려다니게 된다.

### Do: 컴포넌트 합성으로 전달 경로 줄이기

중간 컴포넌트가 상태를 알 필요 없다면 상태를 사용하는 컴포넌트를 상위에서 조립해서 넘긴다.

```tsx
const OrderPage = () => {
  const [selectedOrderId, setSelectedOrderId] = useState<number | null>(null);

  return (
    <OrderLayout
      table={<OrderTable onSelectOrder={setSelectedOrderId} />}
      detail={<OrderDetail orderId={selectedOrderId} />}
    />
  );
};

const OrderLayout = ({
  table,
  detail,
}: {
  table: React.ReactNode;
  detail: React.ReactNode;
}) => {
  return (
    <div className="order-layout">
      <div className="order-list">{table}</div>
      <div className="order-detail">{detail}</div>
    </div>
  );
};
```

`OrderLayout`은 레이아웃만 책임지고 선택 상태에 대해 알지 않는다.

### Do: 화면 전용 Context로 의존성 주입 경계 만들기

여러 하위 컴포넌트가 같은 화면 상태를 공유하고 합성만으로 해결하기 어렵다면 화면 단위 Context를 사용한다.

```tsx
type OrderSelectionContextValue = {
  selectedOrderId: number | null;
  selectOrder: (orderId: number) => void;
  clearSelection: () => void;
};

const OrderSelectionContext = createContext<OrderSelectionContextValue | null>(null);

const useOrderSelection = () => {
  const context = useContext(OrderSelectionContext);

  if (!context) {
    throw new Error("useOrderSelection must be used within OrderSelectionProvider");
  }

  return context;
};

const OrderSelectionProvider = ({ children }: { children: React.ReactNode }) => {
  const [selectedOrderId, setSelectedOrderId] = useState<number | null>(null);

  const value = useMemo(
    () => ({
      selectedOrderId,
      selectOrder: setSelectedOrderId,
      clearSelection: () => setSelectedOrderId(null),
    }),
    [selectedOrderId],
  );

  return (
    <OrderSelectionContext.Provider value={value}>
      {children}
    </OrderSelectionContext.Provider>
  );
};
```

```tsx
const OrderPage = () => {
  return (
    <OrderSelectionProvider>
      <OrderLayout />
    </OrderSelectionProvider>
  );
};

const OrderTable = () => {
  const { selectOrder } = useOrderSelection();

  return <BaseGrid rows={[]} onRowClick={(row) => selectOrder(row.id)} />;
};

const OrderDetailPanel = () => {
  const { selectedOrderId, clearSelection } = useOrderSelection();

  if (!selectedOrderId) {
    return null;
  }

  return <OrderDetail orderId={selectedOrderId} onClose={clearSelection} />;
};
```

이 Context는 앱 전역 상태가 아니다. `OrderPage` 하위에서만 유효한 화면 전용 의존성 주입 레이어다.

### Context 사용 기준

| 조건 | 설명 |
| --- | --- |
| 범위가 명확한가? | 특정 page, feature, dialog, table 안에서만 쓰인다 |
| 중간 컴포넌트가 값을 사용하지 않는가? | 단순 전달만 반복되고 있다 |
| 여러 하위 컴포넌트가 같은 상태/행동을 공유하는가? | table, toolbar, detail panel 등이 같은 선택 상태를 쓴다 |
| 전역 store로 올리면 생명주기가 과해지는가? | 화면을 벗어나면 사라져야 하는 상태다 |

### Context Do Not

```tsx
// 앱 전체 Provider에 화면 전용 상태를 추가하지 않는다.
<AppProvider>
  <OrderSelectionProvider>
    <UserProvider>
      <ThemeProvider>
        <App />
      </ThemeProvider>
    </UserProvider>
  </OrderSelectionProvider>
</AppProvider>
```

```tsx
// 모든 주문 관련 상태를 하나의 거대한 Context에 넣지 않는다.
type OrderContextValue = {
  selectedOrderId: number | null;
  searchParams: OrderSearchParams;
  orders: Order[];
  isLoading: boolean;
  createOrder: () => void;
  cancelOrder: () => void;
  printInvoice: () => void;
  updateAddress: () => void;
};
```

Context는 범위가 넓어질수록 Global State처럼 변한다. 화면 전용 Context는 작고 명확한 책임만 가져야 한다.

### 판단 순서

```md
1. 중간 컴포넌트가 그 props를 실제로 사용하는가?
   -> 사용한다면 props 전달이 자연스럽다.

2. 중간 컴포넌트가 단순 전달만 하는가?
   -> 컴포넌트 합성으로 전달 경로를 줄인다.

3. 여러 하위 컴포넌트가 같은 화면 상태를 공유하는가?
   -> 화면 전용 Context를 고려한다.

4. 여러 페이지/기능에서 공유해야 하는 앱 맥락인가?
   -> 그때만 Global State를 고려한다.
```

## 금지 패턴 요약

### 서버 데이터를 전역 store에 복사

```tsx
orderStore.setOrders(response.data.orders);
```

서버 데이터는 React Query 캐시를 원본으로 둔다.

### URL과 Local State 중복 관리

```tsx
const [page, setPage] = useState(Number(router.query.page ?? 1));
```

URL이 원본이면 URL에서 파생해서 사용한다.

### 단순 UI 상태를 전역화

```tsx
modalStore.open("orderDetail");
```

특정 화면이나 컴포넌트에서만 쓰는 UI 상태는 Local State로 둔다.

### Props Drilling을 전역 상태로 우회

```tsx
orderSelectionStore.setSelectedOrderId(orderId);
```

Props Drilling은 먼저 컴포넌트 합성 또는 화면 전용 Context로 해결한다.

## 핵심 문장

상태관리의 핵심은 "어떤 라이브러리를 쓸까"가 아니라 "이 상태의 원본 소유자가 누구인가"를 정하는 것이다.

서버가 원본이면 React Query, URL이 원본이면 Router, 화면이 원본이면 Local/Form State, 앱 맥락이 원본이면 Global State로 관리한다.

Props Drilling은 Global State를 도입하라는 신호가 아니다. 먼저 컴포넌트 합성으로 의존성 전달 경로를 줄이고, 필요할 때만 특정 화면 범위의 Context로 의존성 주입 경계를 만든다.
